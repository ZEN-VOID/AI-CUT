# CONTEXT.md

## Type Map

| failure_or_outcome_type | root_cause_layer | immediate_fix | systemic_prevention | verification_point |
| --- | --- | --- | --- | --- |
| 章级支流写得很热闹，但回不到卷级主线 | task convergence drift | 补 `汇聚动作` 或 `未汇聚任务去向` | 在 planning 模板固定任务关系槽位 | validation 不再把悬空支流误判成一般节奏波动 |
| 卷级支线存在，但没说明它服务哪个部级主任务 | lineage gap | 补 `上承部级主任务` | 在部级固定 `整部任务关系` | 任务从属可上溯到部级 |
| planning 没声明去向，正文也没有给证据 | source truth under-spec | 先回 `2-卷章` 修源 | 固定 `从属 / 汇聚 / 转挂 / 保留开放` 四象限 | 下游不再凭感觉判断支流命运 |

## Repair Playbook

1. 先锁卷级主任务、支线任务和章级任务从属关系。
2. 若 planning 缺 `从属 / 汇聚 / 转挂 / 保留开放` 去向，优先打回 `2-卷章`。
3. 若 planning 已有去向但正文缺证据，再打回 `1-单章叙事起盘` 或 `7-追读力强化`。
4. 报告必须区分“未汇聚但有去向”和“未汇聚且无去向”。

## Reusable Heuristics

- `支线` 不是天然合理，它必须回答“服务谁、怎样回、没回后去哪”。
- `保留开放` 不是免责条款，只有当 planning 明确声明去向时才成立。
- 最容易假通过的情况是：正文看起来一直在推进，但推进的是一个已经和主任务脱钩的副本。
