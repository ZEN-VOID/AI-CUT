# 4-面板思维链与字段主表

本文件承载 `4-面板` 的字段主表、可见思维快照、工具后反思与返工入口真源。

## 模式与对象

| 项目 | 当前合同 |
| --- | --- |
| 默认模式 | `思维链优化设计`；若面板合同整体重建，可短暂借用 `原生思维链设计`，但最终仍回到本文件接口 |
| 目标对象 | `projects/<项目名>/主体/4-面板/` 下的面板主文件、layout sidecar 与下游参照交接说明 |
| 默认模型策略 | 若模型具备 reasoning / thinking 能力，优先高层布局判断 + 可见快照 + Gate；普通模型仅保留少量显式步骤支架 |
| 可见性边界 | 只保留可审计、可回写、可返工的布局化快照；不把完整内部排版过程全文外显 |
| 工具 / 证据依赖 | 强依赖 `2-设计` 稳定版本或 `3-审计` 通过结果，以及必要的下游消费目标 |
| 受保护接口 | 不破坏 `FIELD-SPNL-01` 到 `FIELD-SPNL-04`；不把面板写成第二版主体设计，也不让主文件和 layout sidecar 失配 |

## Think-Think Design Snapshot

### 启发式工作链

- `不可删性`: 删掉这轮面板判断后，下游会失去“基于稳定设计压缩成统一参照面、同时保留人读和结构布局两侧接口”的主体布局真源。
- `最先推动`: 默认先推动 `FIELD-SPNL-03`，再推进 `FIELD-SPNL-01` 与 `FIELD-SPNL-02`；若继承关系和真源稳定度没锁住，禁止先堆版式细节。
- `先砍什么`: 先砍二次设计主体、只做漂亮文案不做 layout、只做 layout 不给人读主文件、审计未关就继续面板化的伪方案。
- `真正比较尺`: 比较的不是哪个面板看起来更热闹，而是哪种布局最忠于稳定设计、最利于 `5-画面 / 6-视频` 直接消费、最少引入真源漂移。
- `立场纠偏`: 当前判断默认站在下游分镜/视频消费者的立场，而不是站在“再做一版更漂亮的设计稿”的立场。
- `工具后反思`: 读取设计稿与审计结果后，先问真源是否稳定、继承链是否唯一、layout 是否真能服务下游，再继续生成面板。
- `落盘门`: 每条保留判断都必须落到 `FIELD-SPNL-*`、`Gate Summary`、`Fail Code` 或 `Rework Entry` 之一；落不到就删。

### 三轴三重裁决

| 裁决角色 | 在 `4-面板` 的具体问题 | 主要落点 |
| --- | --- | --- |
| `方向轴` | 当前面板要怎样让下游快速理解并消费主体设计，而不是重新解释设计本身 | `FIELD-SPNL-01`、`FIELD-SPNL-02` |
| `成立轴` | 面板是否真正继承 `2-设计` 或审计通过版本，而不是悄悄改写主体真源 | `FIELD-SPNL-03` |
| `优选轴` | 在都成立的布局方案里，哪种最利于后续分镜/视频拿来即用，且不增加返工 | `FIELD-SPNL-01`、`FIELD-SPNL-02`、`FIELD-SPNL-04` |
| `硬门禁轴` | 是否出现二次设计、缺 layout sidecar、审计未过就继续、下游交接不唯一等强制否决条件 | `FAIL-SPNL-01` 到 `FAIL-SPNL-04` |

| 收敛层 | 本层先裁什么 | 服务 `field_id` | 本层收窄了什么 | 正确产物 |
| --- | --- | --- | --- | --- |
| `粗裁决` | 当前是否已具备进入面板层的稳定真源，以及应该继承哪一版设计 | `FIELD-SPNL-03` | 收窄掉审计未关、设计未稳却先做布局的误入 | 继承关系与真源稳定度 |
| `细裁决` | 面板主文件与 layout sidecar 应如何分工，才能既可读又可消费 | `FIELD-SPNL-01`、`FIELD-SPNL-02` | 收窄掉只剩文案或只剩布局字段的单侧输出 | 主文件 + layout sidecar |
| `离散裁决` | 面板完成后如何交给 `5-画面 / 6-视频`，且只给一个主入口 | `FIELD-SPNL-04` | 收窄掉参照目标模糊、交接多头并行的半闭环 | 下游交接说明 |

### 可见 / 隐藏分层

| 层级 | 保留内容 | 去向 |
| --- | --- | --- |
| `可见快照层` | 面板主文件判断、layout sidecar 对齐关系、继承说明、下游交接入口 | 进入面板产物、字段表与验证文档 |
| `Gate Summary 层` | 当前阻塞、主导失败码、返工入口、confidence、unknowns、next_entry | 进入 `validation-report.md` 或 sidecar |
| `隐藏推理层` | 被淘汰的版式比较、未采用的修辞润色、过程中的内部排版废案 | 仅留在内部推理，不进可见合同 |

## 工具后反思与 Gate Summary

### 工具后反思

| 触发节点 | 必问反思 | 继续条件 | 不满足时回退 |
| --- | --- | --- | --- |
| 读取 `2-设计 / 3-审计` 结果后 | 当前版本是否已稳定；是否还存在关键失败项会让面板放大缺陷 | 能明确回答“继承哪一版设计，为什么现在可以布局化” | 回 `S3` 或退回上游 |
| 完成主文件与 layout 初稿后 | 两侧是否同轴；是否一边可读但不可消费，或一边有结构却看不懂 | 主文件与 layout 都可被下游直接使用 | 回 `S1-S2` 重写 |
| 准备交给下游前 | 下游到底应该拿什么作为参照入口；是否仍有二次设计痕迹 | `5-画面 / 6-视频` 有唯一主入口，且继承链可追溯 | 回 `S3-S4` 重收束 |

### Gate Summary Contract

| 项目 | 必须回答 |
| --- | --- |
| `status` | `green / yellow / red` 三选一，不能省略 |
| `dominant_fail_code` | 当前最主要的失败码或 `none` |
| `rework_entry` | 若失败，必须指回 `S1` 到 `S4` |
| `confidence` | `0-1` 浮点值，说明当前面板稳定度 |
| `unknowns` | 尚未被设计真源或审计结果解决的缺口，最多 3 条 |
| `next_entry` | 当前建议进入的后续阶段，如 `5-画面`、`6-视频` 或 `none` |

## Field Master

| field_id | 输出位置/字段 | 内容要求 | 默认责任 Step | 质量维度 | 失败码 |
| --- | --- | --- | --- | --- | --- |
| FIELD-SPNL-01 | 面板主文件 | 每个主体域都有可读的面板主文件，能快速传达主体参照重点而不重做设计定稿 | S1 | 可读性 | FAIL-SPNL-01 |
| FIELD-SPNL-02 | layout sidecar | 每个面板都有可消费的布局 sidecar，且与主文件同轴成立 | S2 | 下游可消费性 | FAIL-SPNL-02 |
| FIELD-SPNL-03 | 继承关系 | 面板清楚继承 `2-设计` 或审计通过版本，说明使用的是哪一版真源 | S3 | 真源一致性 | FAIL-SPNL-03 |
| FIELD-SPNL-04 | 下游交接 | 明确给 `5-画面 / 6-视频` 的参照入口、阻塞说明与唯一下一入口 | S4 | 交接稳定性 | FAIL-SPNL-04 |

## Thought Pass Map

| step_id | 聚焦字段 | 核心问题 | 生成动作 | 未达标信号 |
| --- | --- | --- | --- | --- |
| S1 | FIELD-SPNL-01 | 面板对人读者怎样成立，哪些主体参照必须被看见 | 生成面板主文件 | 只有结构无文案，或主文件变成二次设计稿 |
| S2 | FIELD-SPNL-02 | 面板对下游怎样成立，layout 如何与主文件同轴 | 生成 layout sidecar | 只有文案无结构，或 layout 与主文件脱节 |
| S3 | FIELD-SPNL-03 | 面板是否忠实继承设计真源，当前版本是否稳定 | 固定继承说明与真源版本 | 面板越权二次设计，或审计未过继续布局 |
| S4 | FIELD-SPNL-04 | 做完面板后如何交给下游 | 输出下一入口、阻塞与参照目标 | 参照入口模糊，或多头交接 |

## Validation Matrix

| 检查项 | 通过标准 | 失败信号 |
| --- | --- | --- |
| 真源稳定度检查 | 进入面板前已明确继承的设计版本，关键审计问题已关闭或可控 | 审计未过就直接面板化 |
| 双产物对齐检查 | 主文件与 layout sidecar 同轴成立，可被人读与下游同时消费 | 只有一侧可用，或双侧互相冲突 |
| 继承追踪检查 | 面板能回链 `2-设计` 或审计通过版本 | 面板悄悄改写了主体设计 |
| 下游可消费检查 | `5-画面 / 6-视频` 有清晰参照入口，不需再猜布局含义 | 下游仍需重新整理参照面 |
| 反二次设计检查 | 面板只做布局化表达，不重新发明主体设定 | 面板内容比设计稿还多出核心设定 |
| 可见 / 隐藏分层检查 | 可见层只保留布局快照与继承说明，不混入长篇内部排版过程 | 正文充满未采用版式比较 |
| 工具后反思检查 | 关键读取节点后，发生过“是否可布局 / 是否该回退 / 是否改继承版本”的二次判断 | 读完设计就机械做面板 |
