# Lived-In Character Behavior Contract

## Purpose

本共享合同用于修复 AIGC 角色“像在摆拍、只等着表演、所有注意力都堆在脸上”的问题。它不提供固定台词或固定动作模板，而是要求编导和摄影在关键人物 beat 中先找到角色此刻被什么生活压力推动、正在做什么小事、身体如何下意识反应，以及情绪最终落在哪里。

本合同不授权新增剧情事实、对白、人物设定或硬塞无关道具。所有“活人感”行为必须服从上游文本、人物动作链、空间可达性和道具/环境准入。

## Core Check

关键单人 beat 内部可用四项检查：

| axis | question | valid projection |
| --- | --- | --- |
| `character_anchor` | 这是谁，当前姿态、位置、压力和目标是什么 | 角色站/坐/靠/移动状态、与他人的距离、被什么信息或关系推着走 |
| `micro_activity` | 角色此刻正在做哪件低风险、场景内成立的小事 | 收拾、等待、找东西、调整位置、压住声音、停下手中动作等上游可支持行为 |
| `subconscious_response` | 压力进入时，身体哪里先泄露或偏离原动作 | 呼吸、手指、肩颈、重心、声线、停顿、视线落点或动作突然收住 |
| `emotional_landing` | 这个 beat 最后落成什么可见状态 | 手停住、话收回、距离改变、抬头/低头、沉默延长、决定继续/放弃某个动作 |

`micro_activity` 不是为了让人物“忙起来”而随机添加的动作。它必须至少满足一项：来自上游动作或场景任务、能承接当前空间和角色姿态、能服务角色目标/阻碍、能让情绪反应有落点。删掉后人物动作链更顺的“小事”必须删除。

## Multi-Character Action-Reaction Rule

多人 beat 不要求每个人同时表演。默认先指定：

- `action_driver`：谁在做当前会改变局面的动作、说话、移动或选择。
- `reaction_receiver`：谁被这个动作影响，并用停顿、身体、目光、呼吸、距离或低强度动作反应。
- `ambient_participants`：其他人在场但不抢戏，只保留必要的站位、朝向或低强度背景状态。

禁止把每个角色都写成同等强度地皱眉、擦泪、避视、摸物、后退、看向对方。真实多人戏通常是一个人行动，另一个人被动接住、躲开、迟疑或沉默；镜头和表演应保留这种因果关系。

## Relationship To Action-First Continuity

本合同必须与 `action-first-continuity-contract.md` 同时使用：

- 活人感行为先过人物动作链：当前姿态、位置、朝向、动作向量、可达对象和退出状态必须清楚。
- 活人感小事不得绕过道具准入。没有角色互动、关键信息/规则/证据/危险源或必要环境交代的物件，不得为了“真实”被写成焦点、反射、涟漪、碰撞声或情绪落点。
- 关键情绪 beat 仍要动作经济。一个有效小动作加一个下意识反应通常比多组擦拭、抽泣、摸物、避视更稳定。

## Stage Consumption

| stage | required use |
| --- | --- |
| `2-编导` director layer | 在 `director_substance_plan.character_pressure` 和 `performance_engine` 中给出行为动机：角色此刻被什么生活压力/目标/阻碍推动，以及哪类场景内小事可承托该压力。 |
| `2-编导` performance layer | 把 `micro_activity`、`subconscious_response`、`emotional_landing` 投影到既有动作、对白画面、心理反应或内心独白画面；多人戏建立 `action_driver / reaction_receiver`。 |
| `4-摄影` | 镜头先服务动作-反应因果：谁行动、谁反应、谁只在背景维持状态。不要为了同时照顾所有角色而制造奇怪角度，也不要让所有角色同时“演满”。 |

## Failure Codes

| fail_code | symptom | repair |
| --- | --- | --- |
| `FAIL-LIVED-IN-01` | 角色只摆表情、等台词或展示五官，缺少当下行为任务 | 回到角色压力和当前空间，补一个上游可支持的低风险小事，或承认该 beat 应保持安静但写清情绪落点 |
| `FAIL-LIVED-IN-02` | 多人场面每个人都在同等强度表演，因果关系混乱 | 指定 `action_driver` 和 `reaction_receiver`，其余角色降级为站位/朝向/低强度背景状态 |
| `FAIL-LIVED-IN-03` | 为了真实感加入随机忙动作、无关生活细节或无互动道具 | 删除；若必须保留，回指上游压力、空间任务或角色策略，并通过动作链与道具准入 |
| `FAIL-LIVED-IN-04` | beat 有动作但没有情绪落点，下游不知道下一镜/下一组从何承接 | 补退出姿态、注意力落点、身体接触、距离变化或沉默/声线余波 |

## Review Questions

- 角色是不是只在展示脸、皮肤、光影、发丝或模板表情，而没有当前行为任务？
- 这个小动作是否来自场景、目标、阻碍或生活压力，而不是为了显得“真实”随机添加？
- 多人戏是否清楚谁行动、谁反应、谁只是维持背景状态？
- 如果删除某个小动作或物件细节，人物动作链是否更顺？若是，删除或降级。
- 当前 beat 的最后状态是否能交给下一镜、下一分镜组或下游视频生成？
