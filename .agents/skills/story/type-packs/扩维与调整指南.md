# type-packs 扩维与调整指南

## Purpose

本文件是当前 `type-packs/` 重构后的维护 runbook。

它不再假设目录下存在旧版平铺 contract 文件与 pack 侧车结构。

当前真源已经改成：

- 根目录保留维护说明
- 题材知识真源集中在 `type-packs/网文/`
- 系统 pack 的别名、stage projection 与跨阶段 bias 真源集中在 `type-packs/pack-catalog.yaml`
- 每个题材默认使用“目录包裹 + 同名主文件”或“family craft 目录”结构

本文件只回答：

- 现在应怎样新增/调整题材目录
- 哪些引用逻辑必须同步
- 哪些旧结构假设已经失效

## Current Canonical Structure

### 1. 入口题材目录

形态：

```text
type-packs/网文/修仙/修仙.md
type-packs/网文/末世/末世.md
type-packs/网文/豪门总裁/豪门总裁.md
```

规则：

- 目录名就是题材标识
- 同名主文件是该题材的入口知识文件
- 后续如需补充细化，可继续在同目录增加更多 `.md`

### 2. family craft 目录

形态：

```text
type-packs/网文/狗血言情/*.md
type-packs/网文/古言剧/*.md
type-packs/网文/现实题材/*.md
type-packs/网文/玄幻剧/*.md
type-packs/网文/侦探剧/*.md
type-packs/网文/知乎短篇/*.md
```

规则：

- 这些目录不是单题材入口，而是共享 craft 家族
- 目录内多文件共同构成该 family 的知识源
- 入口题材可以回指一个或多个 family 作为细节补充

### 3. 系统级 pack catalog

形态：

```text
type-packs/pack-catalog.yaml
```

规则：

- 这里维护系统 pack 的 `aliases / semantic_tags / planning_projection / stage_projection / cards_projection / source_dirs`
- 若一个 pack 需要被 `planning / drafting / validation / cards / review` 多阶段共同识别，必须先在这里声明
- 目录题材知识仍以 `网文/<题材>/` 为准；catalog 只负责“如何让系统理解并消费这些 pack”

## Reading Logic

### 当前加载顺序

针对 `type-packs/网文/`，当前推荐读取顺序固定为：

1. 先读 `type-packs/pack-catalog.yaml`，确认 pack alias、projection 与 source_dirs
2. 再命中入口题材目录
3. 若目录下存在同名主文件，先读同名主文件
4. 若还需要细化 craft，再按题材所属 family 追加读取对应 family 目录下的 `.md`

默认题材命中规则：

- 若题材名与 `type-packs/网文/<题材>/` 目录同名，除非特别说明，否则默认该目录就是入口。
- family 目录不是默认主入口，只是按设定补读的细分知识层。

例子：

- `修仙`
  - 先读 `type-packs/网文/修仙/修仙.md`
  - 再按需读 `type-packs/网文/玄幻剧/*.md`
- `豪门总裁`
  - 先读 `type-packs/网文/豪门总裁/豪门总裁.md`
  - 再按需读 `type-packs/网文/狗血言情/*.md`
- `抗战谍战`
  - 先读 `type-packs/网文/抗战谍战/抗战谍战.md`
  - 再按需读 `type-packs/网文/现实题材/*.md`

### 已失效的旧读取假设

以下旧假设现在都不成立：

1. 平铺配置文件是唯一 pack 真源
2. 单一索引侧车是唯一引用入口
3. 单一映射表是唯一映射入口
4. 旧版平铺 contract 文件仍是运行时唯一合同

如果代码或文档还在这样写，视为 stale reference。

## Fast Path Matrix

| 变更类型 | 先改哪里 | 条件性改动 | 一般不要先改 |
| --- | --- | --- | --- |
| 新增一个网文题材 | `type-packs/网文/<题材>/<题材>.md` | 若要接入 family，再补 family 映射说明 | 根 skill 长文案 |
| 给某题材补细节 | 对应题材目录下新增 `.md` | 若内容具有跨题材共享性，改成 family 目录 | 直接改其他兄弟题材 |
| 新增一组共享 family craft | `type-packs/网文/<family>/*.md` | 同步更新本指南中的 family 说明 | 先给每个题材复制一份 |
| 新增一个跨阶段系统 pack / alias pack | `type-packs/pack-catalog.yaml` | 若需要目录知识，再补 `source_dirs` 或新增题材目录 | 直接在脚本里硬编码 pack 分支 |
| 调整某题材引用哪个 family | 先改读取说明 / loader / guide | 若已有测试，再同步改测试 | 只改目录名不改引用点 |
| 重命名题材目录 | 目录 + 全仓引用扫描同步 | loader / 文档 / 说明一起改 | 只挪文件本体 |

## Standard Procedures

### 1. 新增入口题材

最短操作：

1. 新建目录：
   - `type-packs/网文/<题材>/`
2. 新建同名主文件：
   - `type-packs/网文/<题材>/<题材>.md`
3. 至少写清：
   - 核心卖点
   - 冲突引擎
   - 回报模式
   - 禁写模式
   - 若需要，回指哪个 family craft
4. 如果希望 `1-题材选型` 默认能直接命中它，目录名应与实际题材命名保持一致。

### 2. 新增 family craft

最短操作：

1. 新建目录：
   - `type-packs/网文/<family>/`
2. 将该 family 的多份细节 `.md` 放进去
3. 在本指南中登记它属于共享 family，而不是单题材入口

### 3. 新增或调整系统 pack

最短操作：

1. 在 `type-packs/pack-catalog.yaml` 中新增或调整 pack 条目
2. 至少写清：
   - `semantic_tags`
   - `source_dirs`（若需要目录知识）
   - `planning_projection`
   - `stage_projection`
   - `cards_projection`（若 cards 也要差异化）
3. 若它是旧 pack 名的兼容入口，优先写入 `aliases`
4. 若 consumer 已有测试，必须同步补 resolver / validation / cards 等回归

### 3. 目录包裹迁移

适用：

- 原来是 `网文/修仙.md`
- 现在改成 `网文/修仙/修仙.md`

最短操作：

1. 先创建同名目录
2. 把原主文件移入目录
3. 全仓扫描旧路径引用
4. 更新所有文档/脚本中的旧裸路径

不要只做第 2 步。

## No-Silent-Duplication Rules

以下是最常见错误：

1. 同一个题材既在入口目录有一份完整说明，又在 family 目录再复制一份完整说明
2. family craft 被多个题材各自复制，而不是共享引用
3. 文档和代码仍然假定存在旧版平铺配置文件，但目录里已经只有知识型目录结构
4. 只更新 [扩维与调整指南.md](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/type-packs/扩维与调整指南.md)，不更新加载合同和根 skill 回指
5. 在 `validation_runner.py`、`cards_coverage_validator.py` 或其他 consumer 里直接追加新的 pack 名硬编码，却不先更新 `pack-catalog.yaml`

正确做法：

- 题材入口信息放题材目录
- 共享 craft 放 family 目录
- 引用逻辑统一由加载合同和 loader 说明
- 任何结构迁移都必须同步回指路径
- 任何跨阶段差异化 pack 都先写 catalog，再让 consumer 读取 catalog

## Minimal Checklist

做完变更后至少自检：

1. 新题材是否已包在同名目录内
2. 是否存在同名主文件
3. 若依赖 family craft，是否已在文档/loader 中声明
4. 旧路径引用是否已全仓扫描
5. 根 `SKILL.md`、`_shared/type-pack-loading-contract.md`、本指南是否仍然说的是当前结构
6. 若新增或重命名系统 pack，`pack-catalog.yaml`、resolver 和对应 tests 是否同步

## Bottom Line

当前 `type-packs/` 已经不是旧版平铺配置体系。

以后如果有人问：

“这里要怎么扩一个新题材？”

标准答案是：

1. 先在 `网文/` 下建同名目录
2. 放入同名主文件
3. 若需要共享 craft，再挂到 family 目录
4. 若要让系统跨阶段识别它，再补 `pack-catalog.yaml`
5. 最后同步所有引用逻辑

不要再按旧版平铺配置 + 单一侧车索引的思路扩。
