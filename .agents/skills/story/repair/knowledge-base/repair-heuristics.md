# Repair Heuristics

## High-Leverage Checks

- When feedback is vague, classify the symptom before choosing a repair owner. "This chapter feels wrong" is not yet a patch target.
- First search the old term, not the new term. Old-term distribution reveals the true repair surface.
- If a repair touches clue logic, inspect both the first planting and the planned payoff.
- If a repair touches an object, inspect its card before its prose.
- If a repair touches a chapter after validation, inspect stage acceptance packet and return actualization before writing.
- If a repair touches draft or polish prose, route creative rewriting back to the owning stage root skill and record the execution environment only as evidence.

## Common Traps

- Treating a local finding as a local file edit.
- Treating a vague reading complaint as permission for whole-chapter rewriting.
- Updating chapter prose while leaving chapter planning unchanged.
- Updating planning while leaving stage handoff constraints stale.
- Forgetting same-layer predecessors; the current chapter often depends on a clue planted earlier.
- Letting a review report become the only record of a new canonical fact.
