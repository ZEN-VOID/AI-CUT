# 2-设计思维链与字段主表

本文件承载 `2-设计` 的字段主表、可见思维快照、工具后反思与返工入口真源。

## 模式与对象

| 项目 | 当前合同 |
| --- | --- |
| 默认模式 | `思维链优化设计`；若设计合同整体重建，可短暂借用 `原生思维链设计`，但最终仍回到本文件接口 |
| 目标对象 | `projects/<项目名>/主体/2-设计/` 下的设计卡、设计 JSON 与 thinking sidecar 真源，服务 `3-审计 / 4-面板 / 5-画面 / 6-视频` |
| 默认模型策略 | 若模型具备 reasoning / thinking 能力，优先高层设计判断 + 可见快照 + Gate；普通模型仅保留少量显式步骤支架 |
| 可见性边界 | 只保留可审计、可回写、可返工的设计快照；不把完整内部推理全文暴露为主合同 |
| 工具 / 证据依赖 | 强依赖 `1-清单` 的清单与 bridge sidecar，必要时回查 `3-明细` 与 `2-组间` handoff |
| 受保护接口 | 不破坏 `FIELD-SDES-01` 到 `FIELD-SDES-04`；不越权回头重做主体抽取，也不把三件套退化为单一 Markdown 或单一 JSON |

## Think-Think Design Snapshot

### 启发式工作链

- `不可删性`: 删掉这轮设计判断后，下游会失去“既能人读理解、又能机读消费、还能回链设计理由”的主体真源。
- `最先推动`: 默认先推动 `FIELD-SDES-01` 与 `FIELD-SDES-02`；若设计卡与 JSON 尚未同轴成立，禁止先扩写 `thinking sidecar` 或 handoff 包装。
- `先砍什么`: 先砍脱离 bridge 的自由发挥、只有文案没有结构、只有 JSON 没有人读语义、把 thinking sidecar 写成内部流水账的伪设计。
- `真正比较尺`: 比较的不是哪种描述更华丽，而是哪种设计最忠于 bridge、最稳地支持审计/面板/分镜消费、最少引入下游返工。
- `立场纠偏`: 当前判断默认站在 `3-审计 / 4-面板 / 5-画面 / 6-视频` 的消费立场，而不是站在单次展示稿的文案立场。
- `工具后反思`: 读取 bridge 与上游约束后，先问当前设计是否仍忠于主体真源、卡与 JSON 是否一致、是否需要补证据再继续。
- `落盘门`: 每条保留判断都必须落到 `FIELD-SDES-*`、`Gate Summary`、`Fail Code` 或 `Rework Entry` 之一；落不到就删。

### 三轴三重裁决

| 裁决角色 | 在 `2-设计` 的具体问题 | 主要落点 |
| --- | --- | --- |
| `方向轴` | 当前主体最终应怎样被下游稳定继承，哪些设计信息必须进入人读卡与机读 JSON | `FIELD-SDES-01`、`FIELD-SDES-02` |
| `成立轴` | 设计是否被 `1-清单` bridge、脚本证据与上游 handoff 共同支撑，而不是临时猜测 | `FIELD-SDES-01`、`FIELD-SDES-02`、`FIELD-SDES-03` |
| `优选轴` | 在都成立的候选里，哪一种设计最利于后续审计、面板化与下游执行 | `FIELD-SDES-03`、`FIELD-SDES-04` |
| `硬门禁轴` | 是否出现只有单侧产物、与 bridge 冲突无说明、thinking sidecar 空泛、下一入口模糊等强制否决条件 | `FAIL-SDES-01` 到 `FAIL-SDES-04` |

| 收敛层 | 本层先裁什么 | 服务 `field_id` | 本层收窄了什么 | 正确产物 |
| --- | --- | --- | --- | --- |
| `粗裁决` | 这个主体应怎样被设计成稳定真源，哪些输入必须继承、哪些不得越权改写 | `FIELD-SDES-01` | 收窄掉脱离 bridge 的自由发挥与只会堆砌修辞的伪设计 | 设计卡主判断 |
| `细裁决` | 人读卡与机读 JSON 应如何同轴表达，哪些字段是下游真正需要的 | `FIELD-SDES-02`、`FIELD-SDES-03` | 收窄掉一边完整一边空壳、设计理由不可追因的半成品 | 设计 JSON + thinking sidecar |
| `离散裁决` | 本轮应交给审计、面板还是直接下游，以及为何如此路由 | `FIELD-SDES-04` | 收窄掉“交给后面再说”、没有唯一下一入口的假闭环 | 交接说明与下一入口 |

### 可见 / 隐藏分层

| 层级 | 保留内容 | 去向 |
| --- | --- | --- |
| `可见快照层` | 设计卡主判断、设计 JSON 对齐关系、thinking sidecar 的设计理由、下游 handoff | 进入设计产物、字段表与验证文档 |
| `Gate Summary 层` | 当前阻塞、主导失败码、返工入口、confidence、unknowns、next_entry | 进入 `validation-report.md` 或 sidecar |
| `隐藏推理层` | 被淘汰的设计分支、未采用的风格比较、过程中的试写废案 | 仅留在内部推理，不进可见合同 |

## 工具后反思与 Gate Summary

### 工具后反思

| 触发节点 | 必问反思 | 继续条件 | 不满足时回退 |
| --- | --- | --- | --- |
| 读取 `1-清单 bridge / 3-明细 / 2-组间 handoff` 后 | 当前设计输入是否足够；是否存在 bridge 缺口或上游冲突会直接影响设计 | 能明确回答“这个主体为何这样设计，而不是另一种方向” | 回 `S1` 补输入或暂停 |
| 完成设计卡与 JSON 初稿后 | 卡与 JSON 是否同轴；是否有一边完整、一边空壳 | 人读与机读两侧都可直接消费，且不互相打架 | 回 `S1-S2` 重写 |
| 准备写 thinking sidecar 与 handoff 前 | 当前设计解释是否真的能支撑审计与下游，而不是复述表面文案 | sidecar 能回链设计理由，且下一入口唯一 | 回 `S3-S4` 重收束 |

### Gate Summary Contract

| 项目 | 必须回答 |
| --- | --- |
| `status` | `green / yellow / red` 三选一，不能省略 |
| `dominant_fail_code` | 当前最主要的失败码或 `none` |
| `rework_entry` | 若失败，必须指回 `S1` 到 `S4` |
| `confidence` | `0-1` 浮点值，说明当前设计稳定度 |
| `unknowns` | 尚未被 bridge 或上游真源解决的缺口，最多 3 条 |
| `next_entry` | 当前建议进入的下一阶段，如 `3-审计`、`4-面板`、`5-画面`、`6-视频` 或 `none` |

## Field Master

| field_id | 输出位置/字段 | 内容要求 | 默认责任 Step | 质量维度 | 失败码 |
| --- | --- | --- | --- | --- | --- |
| FIELD-SDES-01 | 设计卡 | 每个主体都有稳定的人读设计卡，且能说明其如何继承 bridge 而不是重新发明 | S1 | 可读性 | FAIL-SDES-01 |
| FIELD-SDES-02 | 设计 JSON | 每个主体域都有机读侧车 JSON，且字段与设计卡同轴、不互相冲突 | S2 | 下游可消费性 | FAIL-SDES-02 |
| FIELD-SDES-03 | thinking sidecar | 每个主体都有设计解释层，说明为何这样设计、继承了什么、没有采用什么 | S3 | 设计可追因性 | FAIL-SDES-03 |
| FIELD-SDES-04 | 交接说明 | 能明确下一步进入审计、面板或下游阶段，并说明当前阻塞与唯一下一入口 | S4 | 续跑稳定性 | FAIL-SDES-04 |

## Thought Pass Map

| step_id | 聚焦字段 | 核心问题 | 生成动作 | 未达标信号 |
| --- | --- | --- | --- | --- |
| S1 | FIELD-SDES-01 | 主体最终应怎样被人读地理解，哪些输入不得丢失 | 生成设计卡并锁定继承边界 | 只剩字段堆砌，或脱离 bridge 自由发挥 |
| S2 | FIELD-SDES-02 | 下游怎样机读消费该主体，卡和 JSON 如何同轴 | 生成设计侧车 JSON 并对齐设计卡 | 只能人工理解，或 JSON 与卡口径不一致 |
| S3 | FIELD-SDES-03 | 为什么这样设计而不是那样，哪些判断需要保留为设计解释层 | 写 thinking sidecar | 无设计解释层，或写成内部流水账 |
| S4 | FIELD-SDES-04 | 下一步该进入哪一条链，谁来接手 | 写交接说明、阻塞与唯一下一入口 | 下游入口模糊，或“后续再说” |

## Validation Matrix

| 检查项 | 通过标准 | 失败信号 |
| --- | --- | --- |
| bridge 忠实度检查 | 设计能回链 `1-清单` bridge，且重要约束未丢失 | 设计脱离 bridge，自行发明主体 |
| 卡/JSON 对齐检查 | 设计卡与 JSON 同轴成立，可被人读与机读同时消费 | 只有一侧完整，或双侧互相冲突 |
| thinking sidecar 可追因检查 | sidecar 解释设计理由、继承边界与未采用方案 | sidecar 只有口号或流水账 |
| 双消费面检查 | `3-审计 / 4-面板 / 5-画面 / 6-视频` 至少能直接消费核心设计信息 | 下游仍需重新整理设计真源 |
| handoff 闭环检查 | 下一入口唯一，阻塞与继续条件明确 | 交接只有“继续优化”之类空话 |
| 可见 / 隐藏分层检查 | 可见层只保留设计快照与解释骨架，不混入长篇过程推理 | 文档堆满未采用设计废案 |
| 工具后反思检查 | 关键输入读取后，发生过“是否继续 / 是否补证据 / 是否改设计”的二次判断 | 读完 bridge 就直接机械出稿 |
