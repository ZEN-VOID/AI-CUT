# 主体图复用与状态变体规则

本共享规则适用于 `3-主体/角色/3-生成`、`3-主体/道具/3-生成` 与 `3-主体/场景/3-生成`。它只定义主体图生成前的资产复用、画布上传和状态变体分流，不替代各叶子 `SKILL.md` 的输入、命名、review 或输出合同。

## Preflight Scope

每次真实生图或 prompt-only 规划前，必须先扫描当前项目主体目录：

- `projects/aigc/<项目名>/3-主体/角色/3-生成/`
- `projects/aigc/<项目名>/3-主体/道具/3-生成/`
- `projects/aigc/<项目名>/3-主体/场景/3-生成/`
- `projects/aigc/<项目名>/3-主体/**/design-manifest.yaml`
- 已有同名 JSON 中的 `subject_id`、`subject_name`、`output_stem`、`state_variant_label`、`generation_assets`

匹配键优先级：

1. `subject_id`
2. 规范化 stem：`<主体ID>-<主体名称>-主图`、`<主体ID>-<主体名称>-多视图`
3. `subject_name` 与上游设计文档路径
4. manifest 中的 `generation_assets`

## Same Subject Same State

如果同一主体、同一状态已经存在相关主体图：

- 不重复生成 Midjourney 图像。
- 若目标 libTV 画布已有同名图片节点，直接复用该节点。
- 若本地 canonical 目录已有同名图，项目本地侧视为已满足，不再下载或复制；只在目标 libTV 画布缺同名节点时，用 `libtv upload "<节点名>" -p <canvas_uuid> -f <local_path>` 上传到目标画布。
- 若只有非 canonical 本地文件存在，先复制到对应 canonical 目录并保持同 stem；目标画布缺同名节点时再上传该 canonical 文件。
- JSON、manifest 或执行报告记录：
  - `asset_reuse_decision: "reuse_existing_asset"`
  - `generation_skipped: true`
  - `existing_asset_path`
  - `canvas_action: "node_already_present" | "uploaded_existing_image_to_canvas"`
  - `local_sync_status: "already_present" | "copied"`
  - `libtv_node_name`

不得因为本轮是新集数就为同一主体同一状态重生主体图。

## Same Subject New State

只有识别到同一主体出现新的明确视觉状态时，才允许重新生成状态变体。状态包括但不限于：

- 角色：服装变化、年龄阶段、受伤、战斗前后、伪装、身份切换。
- 道具：开合状态、破损、修复、染血、升级、激活/失效。
- 场景：白天/夜晚、季节、战后废墟、修复后、雨雪雾、火灾前后。

状态变体必须遵守：

- 模型改用 `Lib Image`，不得使用默认 `Midjourney V8.1`。
- 执行前用 `libtv model search --type image "Lib Image"` 与 `libtv model <modelKey>` 解析实际 key/schema。
- 必须以既有同主体图为参考。若参考只在本地，先上传为画布节点；若无参考图，阻断并报告 `missing_base_reference_for_state_variant`。
- 变体 prompt 只描述状态差异和保真约束，保持主体身份、风格、比例、尺寸和材质一致。
- 变体资产命名在主体名称后追加状态后缀：`<主体ID>-<主体名称>-<状态后缀>-主图`，多视图同理为 `-多视图`。
- JSON、manifest 或执行报告记录：
  - `asset_reuse_decision: "generate_state_variant"`
  - `generation_model_policy: "lib_image_state_variant"`
  - `variant_model_display_name: "Lib Image"`
  - `variant_model_key`
  - `state_variant_label`
  - `state_variant_suffix`
  - `base_reference_node_name`
  - `base_reference_asset_path`
  - `style_lock: "inherit_from_base_asset"`
  - `aspect_ratio_lock`

## New Subject Default

只有在找不到同主体同状态可复用资产，且不是状态变体时，才进入默认新主体生成：

- 默认模型：`Midjourney V8.1`。
- 画布节点：libTV `image` 节点。
- 生成数量：遵循真实解析到的 libTV 模型 schema；当前 `mj-v8.1` schema 暴露 `count: [4]` 时，不得强制写入 `count=1`。交付仍按一个 canonical 主图 stem 和一个 canonical 多视图 stem 管理，不能把候选图扩成多个主体。
- 后缀：从 `midjourney风格参数.yaml` 读取对应主体类型比例，并固定追加 `--hd --style raw`。
- 节点名与本地 stem 必须一致。

## Canvas To Project Persistence

无论主体参照图是在第几集画布上生成，只要该图是当前项目的角色、道具或场景主体参照图，完成条件都是“项目本地 canonical 目录存在一份同 stem 资产”。这包含本地已有跳过：本地 canonical 已有时不再下载或复制，只确认并记录 `already_present`。

- 角色：`projects/aigc/<项目名>/3-主体/角色/3-生成/`
- 道具：`projects/aigc/<项目名>/3-主体/道具/3-生成/`
- 场景：`projects/aigc/<项目名>/3-主体/场景/3-生成/`

执行规则：

- 判定顺序固定为：先查项目本地 canonical 目录，再查目标第 N 集画布。
- 本地 canonical 已有同主体同状态资产时：跳过生成，跳过本地下载/复制；若画布缺同名节点则上传本地图，若画布已有则直接复用；记录 `local_sync_action: "confirm_local_canonical_present"` 与 `local_sync_status: "already_present"`。
- 本地 canonical 缺失、但第 N 集画布已有同名主体图片节点时：跳过生成，用 `libtv download -p <canvas_uuid> -n <node_id_or_node_name> -o <对应3-生成目录>` 补齐本地副本；记录 `local_sync_action: "download_existing_canvas_node"` 与 `local_sync_status: "synced"`。
- 本地 canonical 与画布都缺失时：新主体或状态变体在 libTV 画布 `image` 节点生成成功后，执行 `libtv download -p <canvas_uuid> -n <node_id_or_node_name> -o <对应3-生成目录>`，并把下载结果规范化为与画布节点 display name 相同的 stem；记录 `local_sync_action: "download_generated_canvas_node"` 与 `local_sync_status: "synced"`。
- 只有非 canonical 本地文件可复用时：先复制到对应 canonical 目录并保持同 stem；若画布缺同名节点，再上传 canonical 文件；记录 `local_sync_action: "copy_existing_local_to_canonical"` 与 `local_sync_status: "copied"`。
- 本规则当前只约束最终要交付的主图；自 2026-06-19 起多视图默认取消，不再作为主体图片生成的交付项、缺口或完成门。若未来单独恢复多视图，才按同 stem canonical 规则重新纳入。
- prompt-only 模式不得伪造本地图路径；只能记录 `local_sync_status: "pending"` 与阻断原因。

JSON、manifest 或执行报告记录：

- `local_sync_required: true`
- `local_sync_action: "confirm_local_canonical_present" | "download_generated_canvas_node" | "download_existing_canvas_node" | "copy_existing_local_to_canonical" | "prompt_only_pending"`
- `local_sync_status: "already_present" | "synced" | "copied" | "pending" | "blocked"`
- `local_asset_path`
- `download_command`（仅在实际执行下载时必填；`already_present` / `copied` 可为空或 `not_applicable`）
- `download_stdout_path`

## Review Gate

任何主体生成交付前必须能回答：

- 是否扫描了 `projects/aigc/<项目名>/3-主体` 下已有主体资产？
- 同主体同状态是否被跳过或上传复用？
- 只有本地资产时是否上传到了当前目标画布？
- 本地 canonical 已有时是否跳过了下载/复制并记录 `already_present`？
- 画布上新生成或已有但本地缺失的主体图，是否已用 `libtv download` 补齐到对应 `3-生成` 目录？
- 同主体新状态是否使用 `Lib Image` 和既有参考图，而不是 `Midjourney V8.1`？
- 状态变体命名是否包含状态后缀，并记录了参考图、风格锁定和尺寸/比例继承？

缺少以上证据时不得判定生成阶段 pass。
