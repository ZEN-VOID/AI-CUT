# /github-push

用于在已存在远端的前提下，把当前本地正式改动干净地同步到 GitHub，包括清理一次性产物、提交、推送和最小验证。

## 何时使用

- 需要把当前本地改动快速、干净地同步到 GitHub。
- 需要在提交前清理一次性测试产物与调试文件。
- 需要在推送后做最小成本的本地/远端一致性校验。
- 当前仓库已经存在远端与上游关系，且本轮目标不是新建或恢复远端绑定。

## 输入

- 提交信息：
  - 用户提供时直接使用
  - 未提供时自动生成时间戳消息

## Mandatory Gates

- 若 `git remote -v` 为空，立即停止；不得在 `/github-push` 流程内创建、恢复或猜测远端 `origin`。
- 若当前分支未配置上游且用户未明确要求建立跟踪关系，立即停止并要求用户显式确认。
- `/github-push` 只负责“已有远端上的同步”，不负责“建仓/绑仓/重连远端”。

## 标准流程

1. 先运行 `git status --short`，识别本次任务中产生的临时测试脚本、调试探针、临时夹具等非正式交付物。
2. 删除不应进入版本库的一次性产物；若文件可能需要复用，则先判断是否应迁移到正式位置而不是直接删除。
3. 优先运行 `python3 scripts/dev_artifact_cleanup.py` 做仓库级收尾清理。
4. 若仓库不存在 `scripts/dev_artifact_cleanup.py`，则改为手工执行一次目标化清理，并在结尾显式报告：
   - 哪些临时产物已删除
   - 哪些未跟踪文件被判定为正式资产而保留
   - 为什么本轮未走统一清理脚本
5. 执行 `git add .` 暂存本次正式改动。
6. 使用用户提供的提交信息，或生成带时间戳的默认提交信息。
7. 检测当前分支、已声明远端与上游后，执行显式推送命令。
8. 推送后执行最小校验，确认本地工作区与远端分支是否已对齐。

## 验证

- `git status -sb` 无额外挂起文件改动
- 当前分支存在且已成功推送
- 本地分支哈希与 `origin/<branch>` 哈希一致

## 稳定经验

- `push` 流程的核心价值不是“更快”，而是“快但不把垃圾带上远端”。
- 未跟踪文件不应默认视为待清理对象；应先区分缓存/探针噪音与正式迁移产物。
- 当待推送集合里出现运行时缓存噪音时，优先修仓库忽略规则这个“规则源”。
- commit 成功后仍应再次执行 `git status -sb`；若工作区非干净，就补做后续提交后再 push。
- 即便未跟踪集合为空，只要准备执行 `/github-push`，仍应补跑一次仓库清理入口，清掉被 ignore 的运行缓存。
