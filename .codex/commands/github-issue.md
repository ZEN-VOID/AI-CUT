# /github-issue

用于围绕 GitHub Issue 完成“问题理解 -> 根因定位 -> 修复 -> 验证 -> 提交/PR”的闭环。

## 何时使用

- 用户提供 GitHub Issue URL 或编号，希望修复问题并闭环。
- 需要把“问题理解 -> 根因定位 -> 修复 -> 验证 -> 提交/PR”作为标准流程执行。
- 需要确保修复与 Issue 保持可追溯关联。

## 输入

- Issue URL 或 Issue 编号
- 当前仓库上下文
- 对测试、提交、PR 的额外约束

## 标准流程

1. 读取 Issue 详情，确认问题描述、复现步骤、期望行为、影响范围。
2. 在代码库中定位相关文件、调用链、数据流与直接技术原因。
3. 按仓库根因优先原则继续上溯：`Symptom -> Direct Cause -> 规则源 -> 规则源的规则源`。
4. 先修高杠杆源层入口（规则、模板、脚本、校验器、共享 helper）；若仍有必要，再补局部代码修复。
5. 编写或更新测试，运行最相关的验证集，确认修复有效且无明显回归。
6. 生成与 Issue 关联的提交信息，例如包含 `Fixes #<issue>`。
7. 推送分支并创建 PR，确保 PR 与 Issue 正确关联。

## 验证

- Issue 指向的问题已可复现并被修复
- 相关测试通过，必要的质量检查通过
- 提交与 PR 均正确关联 Issue
- 输出中能够说明根因位置、修复点与验证结果

## 稳定经验

- Issue 修复最容易失真的地方是“看起来修好了”，但没有把规则源修掉；因此必须显式做 layered trace。
- 与其追求一次性大而全测试，不如优先跑最贴近问题表面的最小充分验证，再补必要的更广泛检查。
- 非平凡问题不能只停在局部修补；要优先修复能阻止同类问题再次出现的源层合同。
