# 3-审计思维链与字段主表

本文件承载 `3-审计` 的字段主表、可见思维快照、工具后反思与返工入口真源。

## 模式与对象

| 项目 | 当前合同 |
| --- | --- |
| 默认模式 | `思维链优化设计`；若审计合同整体重建，可短暂借用 `原生思维链设计`，但最终仍回到本文件接口 |
| 目标对象 | `projects/<项目名>/主体/3-审计/` 下的结构化审计报告、单主体审计结论与 `writeback-plan.md` |
| 默认模型策略 | 若模型具备 reasoning / thinking 能力，优先高层证据裁决 + 可见快照 + Gate；普通模型仅保留少量显式步骤支架 |
| 可见性边界 | 只保留可审计、可回写、可返工的失败维度与路由结论；不暴露完整内部评论过程 |
| 工具 / 证据依赖 | 强依赖 `2-设计` 产物、`1-清单` bridge、`3-明细`、`2-组间` handoff，以及必要时父级阶段索引 |
| 受保护接口 | 不破坏 `FIELD-SAUD-01` 到 `FIELD-SAUD-04`；不把审计写成纯主观评论，也不在证据不足时直接放行或定死返工 |

## Think-Think Design Snapshot

### 启发式工作链

- `不可删性`: 删掉这轮审计判断后，链路会失去“先锁证据、再结构化失败维度、再给返工路线与下一入口”的闭环能力。
- `最先推动`: 默认先推动 `FIELD-SAUD-01` 与 `FIELD-SAUD-02`；若输入证据链和失败维度尚未成立，禁止先写修订建议或放行结论。
- `先砍什么`: 先砍审美化点评、无证据 verdict、只有“建议优化”没有失败维度、明知需返工却提前放行的伪审计。
- `真正比较尺`: 比较的不是点评多锋利，而是哪种失败维度最可追溯、最能支撑返工、最能正确决定下一入口。
- `立场纠偏`: 当前判断默认站在 `2-设计` 返工执行者和下游阶段 gate 的立场，而不是站在单次评论稿作者的立场。
- `工具后反思`: 读取设计产物与上游证据后，先问偏差到底违反了哪一层合同、证据是否足够、是否需要补证据再继续裁决。
- `落盘门`: 每条保留判断都必须落到 `FIELD-SAUD-*`、`Gate Summary`、`Fail Code` 或 `Rework Entry` 之一；落不到就删。

### 三轴三重裁决

| 裁决角色 | 在 `3-审计` 的具体问题 | 主要落点 |
| --- | --- | --- |
| `方向轴` | 这次到底在审什么，哪些输入和约束必须进入证据链 | `FIELD-SAUD-01` |
| `成立轴` | 当前偏差是否真的成立，且能回链到 `1-清单 / 2-设计 / 3-明细 / 2-组间` 的约束 | `FIELD-SAUD-01`、`FIELD-SAUD-02` |
| `优选轴` | 在成立的偏差里，哪些应优先返工、哪些可带注意项继续、下一入口该如何选择 | `FIELD-SAUD-03`、`FIELD-SAUD-04` |
| `硬门禁轴` | 是否出现证据不足、失败维度空泛、返工建议不可执行、关键失败未关却放行等强制否决条件 | `FAIL-SAUD-01` 到 `FAIL-SAUD-04` |

| 收敛层 | 本层先裁什么 | 服务 `field_id` | 本层收窄了什么 | 正确产物 |
| --- | --- | --- | --- | --- |
| `粗裁决` | 这次审计到底围绕哪批设计与哪些上游约束展开 | `FIELD-SAUD-01` | 收窄掉“只看当前稿感受”“不补证据也能审”的空裁决 | 审计输入面与证据链 |
| `细裁决` | 偏差到底成立在哪里，哪些失败维度最能支撑返工 | `FIELD-SAUD-02`、`FIELD-SAUD-03` | 收窄掉主观点评、无优先级问题堆、无法回链的模糊缺陷 | 失败维度 + 修订建议 |
| `离散裁决` | 审完后回 `2-设计`、进 `4-面板`、直接放行，还是暂缓继续 | `FIELD-SAUD-04` | 收窄掉“先放过去再说”与没有唯一下一入口的半闭环 | 下一入口与结案条件 |

### 可见 / 隐藏分层

| 层级 | 保留内容 | 去向 |
| --- | --- | --- |
| `可见快照层` | 审计输入束、失败维度、返工建议、下一入口 | 进入审计报告、字段表与 `writeback-plan.md` |
| `Gate Summary 层` | 当前阻塞、主导失败码、返工入口、confidence、unknowns、next_entry | 进入 `validation-report.md` 或审计 sidecar |
| `隐藏推理层` | 被淘汰的轻微问题、未采用的点评措辞、过程中的内部权衡 | 仅留在内部推理，不进可见合同 |

## 工具后反思与 Gate Summary

### 工具后反思

| 触发节点 | 必问反思 | 继续条件 | 不满足时回退 |
| --- | --- | --- | --- |
| 读取 `2-设计 / 1-清单 / 3-明细 / 2-组间` 后 | 当前证据链是否足够；是否知道这次到底在审什么、依据哪一层合同 | 能明确回答“偏差针对什么、证据来自哪里” | 回 `S1` 补证据 |
| 完成失败维度初稿后 | 每条失败维度是否能回链约束；是否存在仅靠主观印象的判断 | 失败维度结构化，且至少能说明一条违反的上游约束 | 回 `S2` 重写 |
| 准备给出下一入口前 | 关键失败项是否已排序清楚；回 `2-设计`、进 `4-面板` 或放行的条件是否唯一 | 下一入口唯一，且返工或放行条件明确 | 回 `S3-S4` 重收束 |

### Gate Summary Contract

| 项目 | 必须回答 |
| --- | --- |
| `status` | `green / yellow / red` 三选一，不能省略 |
| `dominant_fail_code` | 当前最主要的失败码或 `none` |
| `rework_entry` | 若失败，必须指回 `S1` 到 `S4` |
| `confidence` | `0-1` 浮点值，说明当前审计结论稳定度 |
| `unknowns` | 尚未被证据链解决的缺口，最多 3 条 |
| `next_entry` | 当前建议进入的后续阶段，如 `2-设计`、`4-面板`、下游放行或 `none` |

## Field Master

| field_id | 输出位置/字段 | 内容要求 | 默认责任 Step | 质量维度 | 失败码 |
| --- | --- | --- | --- | --- | --- |
| FIELD-SAUD-01 | 审计输入面 | 能定位 `2-设计` 产物与上游证据链，并说明本轮审计范围与缺口 | S1 | 输入完整性 | FAIL-SAUD-01 |
| FIELD-SAUD-02 | 失败维度 | 审计结果是结构化失败维度，不是空泛点评；每条维度都应可回链上游约束 | S2 | 成立性 | FAIL-SAUD-02 |
| FIELD-SAUD-03 | 修订建议 | 每个失败维度都有可执行返工建议，且说明优先级与写回位置 | S3 | 修复针对性 | FAIL-SAUD-03 |
| FIELD-SAUD-04 | 下一入口 | 明确回 `2-设计`、进入 `4-面板`、直接放行或暂缓继续，并给出唯一入口 | S4 | 闭环清晰度 | FAIL-SAUD-04 |

## Thought Pass Map

| step_id | 聚焦字段 | 核心问题 | 生成动作 | 未达标信号 |
| --- | --- | --- | --- | --- |
| S1 | FIELD-SAUD-01 | 这次到底在审什么，证据链是否完整 | 锁定输入束、审计范围与证据来源 | 输入链不全，或审计对象模糊 |
| S2 | FIELD-SAUD-02 | 偏差到底成立在哪里，违反了哪条上游约束 | 生成结构化失败维度与对应证据 | 只有审美评论，没有结构化失败点 |
| S3 | FIELD-SAUD-03 | 应该怎么修，返工要落到哪里 | 生成修订建议、优先级与写回位置 | 建议无法执行，或只会说“优化一下” |
| S4 | FIELD-SAUD-04 | 审完后下一步去哪里，谁来接手 | 输出唯一入口、放行/返工条件与阻塞 | 闭环含混，或关键失败未关却放行 |

## Validation Matrix

| 检查项 | 通过标准 | 失败信号 |
| --- | --- | --- |
| 输入证据链检查 | 已明确设计稿、bridge、脚本与编导等关键输入的来源与缺口 | 证据链缺失仍继续出终局裁决 |
| 失败维度结构化检查 | 每条问题都能落入明确维度，并说明违反的约束 | 只有“感觉不对”“建议优化” |
| 上游回链检查 | 关键结论都能回链到 `1-清单 / 2-设计 / 3-明细 / 2-组间` | 审计结论与上游真源脱节 |
| 修订建议可执行检查 | 每条失败维度都有具体返工动作和写回位置 | 返工建议停留在抽象口号 |
| 下一入口闭环检查 | 回 `2-设计`、进 `4-面板` 或放行只有一个主入口 | 审完后仍然不知道下一步去哪 |
| 可见 / 隐藏分层检查 | 可见层只保留证据化结论，不混入大段评论过程 | 文档里充满长篇主观感想 |
| 工具后反思检查 | 关键读取节点后，发生过“证据够不够 / 结论要不要重判 / 是否该回退”的二次判断 | 读完材料就直接写 verdict |
