# 4-Validation 扩维与调整指南

## Purpose

本文件是 `4-Validation` 技能组面向维护者的快捷操作说明。

用途不是替代规范真源，而是把“未来审计技能调整或新增时，最短该怎么改”整理成一页可执行 runbook。

规范真源仍然是：

1. [`_shared/validation-dimension-registry.yaml`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/validation-dimension-registry.yaml)
2. [`_shared/validation-root-contract.md`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/validation-root-contract.md)
3. [`_shared/validation-child-output-contract.md`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/validation-child-output-contract.md)
4. [`_shared/checker-output-schema.md`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/checker-output-schema.md)

本文件只回答：

- 改什么最少
- 改动顺序是什么
- 哪些地方绝不能静默漏改

## Core Rule

未来新增、删除、改名或重路由 validation 维度时，默认不要从父技能文档开始改。

最稳顺序固定为：

1. 先改 registry
2. 再改对应 child package
3. 最后只在必要时改 schema / call matrix / 示例

一句话：

- `registry` 管维度存在、step hook、终验 mandatory
- `child package` 管该维度如何审
- `schema` 管结构化字段长什么样

## Fast Path Matrix

| 变更类型 | 必改真源 | 条件性改动 | 一般不需要先改 |
| --- | --- | --- | --- |
| 调整某已有维度在哪些 drafting step 做 inline hook | `validation-dimension-registry.yaml` | 若输出字段也变，再改 `checker-output-schema.md` | `3-Drafting` 父技能正文 |
| 调整某已有维度的默认返工节点 | `validation-dimension-registry.yaml` | 若 child 合同描述也变，再改该 child `SKILL.md` | 其他 sibling child |
| 新增一个 validation 维度 | `validation-dimension-registry.yaml` + 新 child package | `checker-output-schema.md`、示例 JSON、call matrix | 直接手写进 7 个 drafting step |
| 删除或停用一个 validation 维度 | `validation-dimension-registry.yaml` | 对应 child package、示例 JSON、schema | 先改父技能长文案 |
| 仅修改维度内的 prose 判据，不改字段形状 | 对应 child `SKILL.md/CONTEXT.md` | 若默认返工 step 变，再改 registry | 其他共享文件 |
| 修改结构化 issue 字段 | `validation-child-output-contract.md` + `checker-output-schema.md` | 示例 JSON、下游消费点 | 只改 child 自己 |

## Standard Procedures

### 1. 调整已有维度

适用：

- `结构兑现` 现在想多挂一个 drafting step
- `逻辑自洽校验` 的默认返工节点要改
- `人物一致性` 不再是终验 mandatory

最短操作：

1. 编辑 [`validation-dimension-registry.yaml`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/validation-dimension-registry.yaml)
2. 修改对应维度下的：
   - `drafting_inline.checkpoints`
   - `default_rework_targets`
   - `final_acceptance.mandatory`
   - `final_acceptance.weight`
3. 如果维度职责边界也变了，再改对应 child `SKILL.md`

通常不需要先改：

- `3-Drafting/SKILL.md`
- `4-Validation/SKILL.md`

因为它们应当只回指 registry，而不是自己再维护一份维度表。

### 2. 新增一个维度

适用：

- 新增 `世界观压力`
- 新增 `场景功能一致性`
- 新增 `视角稳定性`

最短操作：

1. 在 [`validation-dimension-registry.yaml`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/validation-dimension-registry.yaml) 新增一项：
   - `role_id`
   - `dimension`
   - `skill_path`
   - `report_filename`
   - `final_acceptance`
   - `drafting_inline.checkpoints`
   - `default_rework_targets`
   - `upstream_source_owners`
2. 新建对应子技能包：
   - `<新维度>/SKILL.md`
   - `<新维度>/CONTEXT.md`
3. 若该维度需要新的 machine-readable metrics / risk 字段：
   - 同步改 [`checker-output-schema.md`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/checker-output-schema.md)
   - 视需要改 [`validation-child-output-contract.md`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/validation-child-output-contract.md)
4. 若你希望它进入样例：
   - 更新 [`validation.fail-quality.back-to-drafting.example.json`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/examples/validation.fail-quality.back-to-drafting.example.json)

### 3. 删除或停用一个维度

适用：

- 某维度准备废弃
- 某维度只保留终验，不再做 drafting inline

最短操作：

1. 先改 registry
   - 删除整个维度项，或把 `drafting_inline.enabled` / `final_acceptance.mandatory` 改掉
2. 再处理 child package
   - 保留但降级说明，或直接删除
3. 再检查共享投影
   - `checker-output-schema.md`
   - 示例 JSON
   - `claude-code-call-matrix.md`

### 4. 改 issue / packet 字段形状

适用：

- 想给 issue 增加 `evidence_refs`
- 想给 packet 增加 `blocking_scope`

最短操作：

1. 先改 [`validation-child-output-contract.md`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/validation-child-output-contract.md)
2. 再改 [`checker-output-schema.md`](/Volumes/AIGC/AIGC-DREAM-MAKER/.agents/skills/story/4-Validation/_shared/checker-output-schema.md)
3. 再改：
   - 示例 JSON
   - 相关 child `SKILL.md`
   - 若下游要消费，再检查 `review/` 或 `5-Loopback`

不要反过来：

- 先在某个 child `SKILL.md` 里写了新字段
- 却没改共享 contract 和 schema

这会直接长出第二真源。

## No-Silent-Duplication Rules

以下是最常见的错误做法：

1. 在 `3-Drafting` 七个子技能里各自写一份 step hook 维度名单
2. 在 `4-Validation/SKILL.md` 再维护一张手写 validator 列表
3. 在 child `SKILL.md` 引入新的 packet 字段，但不改 shared contract

正确做法：

- step hook 名单统一看 registry
- validator roster 统一看 registry + `validation-team-contract.md`
- 若当前维度需要参与本地自动 runner，还必须同步补 `.agents/skills/story/scripts/validation_runner.py` 里的 `ROLE_RUNNERS` 映射与对应 handler；registry 仍是“跑哪些维度”的真源，runner 只负责“该维度当前如何做 baseline 自动判定”。
- 输出字段统一看 child output contract + checker schema

## Minimal Checklist

做完变更后，至少自检这几项：

1. registry 是否已更新
2. 对应 child package 是否存在且边界清楚
3. 若字段变了，schema 是否同步
4. 若样例受影响，示例 JSON 是否同步
5. 父技能是否仍然只回指 registry，而不是长出第二张维度表

## Bottom Line

以后如果有人问：

“新增一个审计技能包，最少改哪里？”

标准答案就是：

1. 先改 registry
2. 再补 child package
3. 字段变了才改 schema

不要从父技能长文案开始改。
