## Bad Example: 把旧 F1 runtime 当 workflow 主链

```yaml
symptom: 用旧 F1 scripts、EDL、MoviePy 或 ffmpeg filter graph 直接生成 final，再把结果称为 workflow 输出。
why_bad: workflow 的实现底座是 HyperFrames；旧 F1 只能作为迁移背景，不是 runtime 依赖。
fail_code: FAIL-WORKFLOW-HYPERFRAMES-ONLY
rework: 回 N2-HYPERFRAMES-LOAD，重新用 HyperFrames core/cli/media/animation/creative 建立 composition 和验证链路。
```

## Bad Example: 空素材目录被当成证据

```yaml
symptom: projects/素材/收益素材/ 目录为空，但 storyboard 写“已使用收益素材”。
why_bad: 目录存在只代表候选落位；没有真实文件、manifest 或视觉复核，就不能进入 asset_evidence.json。
fail_code: FAIL-ASSET-EVIDENCE
rework: 回 N3-MEDIA-EVIDENCE，读取真实文件或标记素材缺口。
```

## Bad Example: 字幕按总时长平均切分

```yaml
symptom: 报告写 manual_script_audio_duration，字幕看似出现，但没有 per-cue audio_anchor。
why_bad: final 台词字幕必须跟随音频主时钟；总时长平均切分只能作为草稿或 conditional。
fail_code: FAIL-DIALOGUE-CLOCK
rework: 回 N4-DIALOGUE-CLOCK，做 ASR 或逐 cue 人工校时，并运行 strict validator。
```

## Bad Example: 字幕顺序靠 HTML 临时交换

```yaml
symptom: dialogue_alignment.json 里 cue 没有 script_order，HTML 中字幕文本和 data-start 看似存在，但第 2 条字幕显示了第 4 条台词，或 DOM 顺序和音频顺序不一致。
why_bad: 字幕顺序是主时钟合同的一部分；没有 script order、audio anchor 和 data-cue-id 映射，渲染层看似有字幕也无法证明与声音同步。
fail_code: FAIL-DIALOGUE-CLOCK
rework: 回 N4-DIALOGUE-CLOCK 重建按原文顺序推进的 cue 和音频锚点；回 N6 为字幕 DOM 写 data-cue-id 并重新运行 validate_dialogue_sync.py --strict-final。
```

## Bad Example: 文案和音频跨编号错配

```yaml
symptom: `文案12.txt` 的字幕配了 `文案21.mp3`，或把 `BGM.mp4` 当成旁白主时钟。
why_bad: 批量脚本和音频的同名 stem 是主时钟绑定键；跨编号会让字幕语义和声音彻底错位。
fail_code: FAIL-DIALOGUE-CLOCK
rework: 回 N1/N4，重建 script_audio_pair_map，只允许同 stem 配对；BGM 只能进入背景音乐计划。
```

## Bad Example: 批量成片散落在 renders 目录

```yaml
symptom: 10 条 final 分别留在各工程 renders/ 下，日期根和 成片/ 没有统一可查看文件。
why_bad: 用户需要直接查看最终视频；过程文件和 final 交付口径混在一起会破坏批量交付。
fail_code: FAIL-BATCH-FINAL-COLLECTION
rework: 回 N9-CLOSE，将 final 统一归集到 projects/output/<日期>/成片/，并同步 ledger/report。
```

## Bad Example: 社媒广告像动态 PPT

```yaml
symptom: 整片由静态卡片、内部标题、流程标签、图文网格组成，明明有影像/工具素材却没有背景承托。
why_bad: 信息结构清楚不等于视频表达成立；社媒广告默认需要影像动势、字幕同步和素材匹配，不能用额外展示层替代视频表达。
fail_code: FAIL-DECK-LIKE-AD
rework: 回 N3/N5/N6，重建 background_layer 和 dialogue_caption 核心计划；只有用户显式要求时才补 pip_asset 或 editorial_overlay。
```

## Bad Example: 画中画形式回指但 0 分匹配

```yaml
symptom: workflow_assignment.json 里每个 PiP 都有 segment_id，但 match_score=0 或缺 match_terms。
why_bad: segment_id 只是形式引用，不代表真正消费了视频说明或 cue 语义。
fail_code: FAIL-QUANT-VISUAL-CONTRACT
rework: 仅在用户显式要求 PiP 或已有工程必须保留 PiP 时回 N3/N5，补 cue-level match、图片角色、视频 manifest hint 和 match_reason；否则删除 PiP，回到背景视频匹配。
```

## Bad Example: 逐句随机拼素材

```yaml
symptom: 每句文案随机换一个视频或图片，没有爆款开头、内容、引流三段，也没有背景拉通和字幕核心层分工。
why_bad: 文案顺序不是视频节奏；随机素材轮播无法形成广告结构，也无法机械审计画面层次。
fail_code: FAIL-LAYERED-ASSEMBLY
rework: 回 N3/N5，先建立 background_throughline 和 timeline_segments，再投影 background_video/dialogue_caption 核心层并运行 validate_visual_contract.py --strict-social-ad；显式要求时再加 PiP/大字报验证参数。
```

## Bad Example: 网页预览当成成片交付

```yaml
symptom: 任务没有锁定 projects/内容/文案 + projects/内容/音频 的同名路径，过程中自行生成了一段文案/旁白；背景层仍使用蒙版或 opacity；最终只停在网页端预览。
why_bad: 当前内容池的同 stem 文案/音频是内容真源和主时钟；背景层已要求无蒙版、全不透明；workflow 普通完成态是本地 final MP4，不是浏览器预览页。
fail_code: FAIL-DIALOGUE-CLOCK / FAIL-LAYERED-ASSEMBLY / FAIL-RENDER-FINAL
rework: 回 N1/N4 锁定 selected_script_audio_pair 与 source_script/source_audio/script_audio_stem；回 N5/N6/N7 移除背景 mask/opacity；回 N8/N9 渲染并归集本地 canonical MP4。
```

## Bad Example: 开头跳过素材、引流放大、PiP 单小窗

```yaml
symptom: 开头没有使用 projects/素材/开头素材/，或开头素材被 cover 裁掉主体；引流素材本来模糊还被 scale(1.4) 放大；显式 PiP 路线里全片只有一个很小的 PiP 角标。
why_bad: 开头和 CTA 问题会直接伤害观众首屏与清晰度；显式 PiP 也不能只靠“数量大于 0”通过。
fail_code: FAIL-LAYERED-ASSEMBLY / FAIL-PIP-DIVERSITY-MATCH / FAIL-QUANT-VISUAL-CONTRACT
rework: 回 N3/N5/N6，补开头素材 full-frame 5-10 秒证据，引流 no-upscale/native-scale 证据；显式 PiP 路线把 PiP 改成同屏多窗网格，最后跑 validate_visual_contract.py --strict-social-ad，必要时追加 --require-pip。
```

## Bad Example: 左上角标题显示工作流程

```yaml
symptom: 大字报或左上角标题写成“工作流程”“内部学习交流”“项目复盘”，而不是当前文案段落的利益点或结果。
why_bad: final 视频是给短视频平台 C 端用户看的；内部流程标题会把广告表达变成团队说明，也无法证明来自匹配文案 cue。
fail_code: FAIL-EDITORIAL-OVERLAY / FAIL-QUANT-VISUAL-CONTRACT
rework: 回 N1 写 audience_profile=short_video_c_end_user；回 N5 按 source_cue_ids/source_text/match_reason 重提炼 core_title；回 N6 删除观众可见内部流程标题并重新运行 validate_visual_contract.py --strict-social-ad。
```

## Bad Example: 未要求却加大字报和画中画

```yaml
symptom: 用户只要求按素材做成片，但工程默认生成了左上角大字报和多个 PiP 证据窗。
why_bad: 默认展示层合同已经改为 opt-in；未被要求的大字报/PiP 会分散注意力，也会把执行重点从最佳视频匹配、字幕和音画同步上移开。
fail_code: FAIL-EDITORIAL-OVERLAY / FAIL-PIP-DIVERSITY-MATCH
rework: 回 N5/N6 删除未授权 `editorial_overlay` / `semantic_pip`，把计划改为 background_video + dialogue_caption 核心层；如用户补充明确要求，再重新启用对应可选层。
```

## Bad Example: 素材重复播放并覆盖监控历史

```yaml
symptom: 同一个素材在一条 final 里被切成多个片段反复播放，任务完成后又用 rebuild 方式重写 projects/素材使用监控.csv。
why_bad: 单条 final 内同素材只能出现一次；全局 CSV 是历史累计账本，普通收口只能在旧记录上追加本轮已验证 actual usage，不能刷新覆盖。
fail_code: FAIL-ASSET-USAGE-MONITOR / FAIL-QUANT-ASSET-DIVERSITY
rework: 回 N3 读取 CSV 并排除已达 20 次素材；回 N5/N6 换用不同素材路径并删除重复 track；回 N9 用 update_asset_usage_monitor.py 默认 cumulative_add 重跑，确认任一素材累计 <=20。
```
