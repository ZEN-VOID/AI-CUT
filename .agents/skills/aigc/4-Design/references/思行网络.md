# 4-Design 思行网络

本文件为 `aigc/4-Design/SKILL.md` 的骨架细则下沉载体，只承接节点展开、返工语义与 closure 书写提示，不拥有独立真源裁决权。

## Node Playbook

| node_id | 必须回答的问题 | 允许写入的证据 | 禁止行为 |
| --- | --- | --- | --- |
| `N0` | 这是不是 `4-Design` 父层任务 | `stage_scope`、`output_mode` | 直接进入 leaf 或先写 closure |
| `N1` | 当前项目、集数与 runtime 根是什么 | `project_root`、`episode_scope`、`runtime_root` | 未锁路径就推导 leaf 输出 |
| `N2` | 该进哪个 tranche，route mode 是什么 | `route_mode`、`tranche_scope`、`blocked_tranches` | 因为用户提到多个词就默认全链执行 |
| `N3` | 哪些 domain active，哪些 pending | `active_leaf_set`、`pending_leaf_set` | 把 pending 当成 active，或给未命中域补空 |
| `N4` | `roles.supervision` 是否启用、是否采纳 | `advisory_mode`、`advisory_note` | 把 advisory 写成落盘后 reviewer |
| `N5` | leaf 是否真实落盘、路径是否正确 | `executed_leaf_outputs`、`runtime_alignment` | 父层替 leaf 造第二份 canonical |
| `N6` | 当前该 `pass / rework / stop` 吗 | `stage_verdict`、`rework_entry` | 只给抽象评价，不给返工节点 |
| `N7` | 如何把本轮结论一次性收束 | closure 四段、下一入口 | 另起 sidecar 或并列多份结论 |

## Route Semantics

### `single-tranche-single-domain`

- 适用于用户明确只要一个 tranche 下的一个域。
- 父层只调度该域 leaf，不补其他域。
- closure 中必须明确“局部完成”。

### `single-tranche-multi-domain`

- 适用于同一 tranche 下多个 active domain 同时命中。
- 父层可以形成一个 dispatch 集合，但 closure 仍要按域枚举已执行与未执行项。
- 未命中的 domain 仍不得补空。

### `cross-tranche-handoff`

- 适用于用户要求从上游 tranche 推进到下游 tranche，且前置条件已经满足。
- 父层只允许沿 `1-清单 -> 2-设计 -> 3-面板` 顺序 handoff。
- 任一前置条件缺失时，必须停在缺口处，不能假装下游完成。

## Rework Semantics

| 症状 | 默认返工入口 |
| --- | --- |
| tranche 选错或顺序漂移 | `N2` |
| active/pending 判错 | `N3` |
| advisory 越权到收尾 | `N4` |
| leaf 输出错路径或父层造第二真源 | `N5` |
| closure 缺少四段或下一入口 | `N7` |

## Closure Writing Hints

`validation-report.md` 的 `Thinking-Action Closure` 推荐按以下顺序写：

1. `思考过程`
   当前为什么选这个 tranche/domain 组合，为什么没有走其他路。
2. `关键证据`
   本轮调度了哪些 leaf，哪些输出已落盘，哪些 advisory 被采纳或弃用。
3. `风险/例外`
   哪些 sibling 仍 pending-migration，哪些 scope 只是局部完成。
4. `下一入口`
   当前应该回到哪个 tranche/domain，或进入 `5-Image / 6-Video / review` 的哪个入口。
