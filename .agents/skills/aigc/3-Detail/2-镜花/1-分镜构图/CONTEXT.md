## Purpose & Loading Contract

- 本文件保存 `1-分镜构图` 的经验层。
- 命中本 skill 时必须和同目录 `SKILL.md` 一起加载。
- 经验层只沉淀失败类型、修复顺序与可复用 heuristic，不重写 `module-spec.yaml`、叶子 `module-spec.yaml` 或 `SKILL.md` 的规范字段。

## Type Map

| failure_or_outcome_type | immediate_fix | verification_point |
| --- | --- | --- |
| 有镜头感但没有骨架 | 回到 `watermoon_inheritance + shot_count_plan + shot_slot_map` 四件套，先重建组级脊柱，再进入叶子串行 | 后三支能直接承接，不再反向救骨架 |
| `构图形式` 先天发虚 | 回到主体/陪体/背景、空间锚点、轴线和 frame task，先让画面站住 | `composition_skeleton` 能解释“谁压谁、镜头怎么站住” |
| `构图形式` 退化成站位复述 | 先锁一个经典或稳定的构图范式，再补轴线、锚点和 frame task；把 blocking 留给 `shot_slot_map` / `运动表现` | 成稿先能说出是哪类画面组织，而不是只会重说谁站前谁站后 |
| `构图形式` 漏掉留白与安全区 | 回到头顶留白、视线留白、眼睛落点与关键动作安全区，先校准“画面是否可读”再谈风格名 | 重要信息不出安全区，且留白不是随手摆放 |
| 双人关系镜头只有“对话感”没有关系型站姿 | 在 `profile two-shot / front two-shot / OTS / 框中窥视 / 遮挡窥视` 中先锁一型，再解释谁压谁、谁看谁 | 读者能直接判断两人是并置、对抗、窥视还是被隔断 |
| 失衡修辞被滥用 | 只有在危险、精神不稳、压迫、失语、异常时才允许荷兰角、偏轴失衡、过度切脸或强遮挡；否则回到稳定构图 | 成稿中的异常构图都能回答“为什么此刻要不舒服” |
| 构图层没提前照顾连续性 | 在 `构图形式` 先补轴线、银幕左右关系、视线方向与正反打预判，再进入 `镜头类型` | 后续正反打与视线匹配不需要重救 |
| `景别景深` 漂成摄影参数 | 退回“观看距离 + 空间偏置 + 景深层级 + 焦点/景框策略 + 心理距离”五件事，不写焦段、光圈或器材值 | `shot_size_rhythm_preview` 能解释情感距离、画外阅读和焦点组织，而不是器材设置 |
| `镜头类型` 被写成器材或运镜 | 回到 POV、descriptor 与观看姿态，明确它是叶子上位类，内部统领 `镜头类型 / 镜头框架 / 镜头视角`，不是设备选择或运动路线 | `shot_descriptor_lock` 和 `focus_spatial_logic` 可被后续摄影、运镜直接消费 |
| 画面挺好看但没有戏 | 回到 `镜头类型` 叶子，先锁戏剧任务，再把低/高/侧/背/顶/过肩/POV/荷兰角映射到角色状态、情绪落点和叙事站位 | 成稿能解释为什么必须从这个角度看，而不是只会说“更电影” |
| slot 可读但画面不可拍 | 回到 `SHOT-N2-ANCHOR-SLOT` 与 `构图形式`，重查主陪背景、视线路径与空间锚点是否共用同一镜头任务 | `slot_id`、`构图骨架` 与 `视线组织` 能互相解释 |
| `thinking_process` 只剩审美口号 | 回到思行节点证据，逐条补齐输入依据、动作和 patch 对应关系 | branch review 能从 `thinking_process` 直接追到 `patch_payload` |

## Repair Playbook

1. 先查 shared root 的 `分镜切换`、`剧本正文` 与 `水月` 是否稳定，再判断是不是构图层问题。
2. 若本组锚点说不清，先补 `watermoon_inheritance + shot_slot_map`，不要直接改叶子输出。
3. 若画面站不住，优先回退 `构图形式`，不要直接用景别或镜头类型硬救。
4. 若景别景深变成摄影技术语言，立刻收回到观看距离与心理距离，不允许把摄影参数混进构图层。
5. 若开放景框、前景遮挡或群像分层说不清叙事任务，回到 `景别景深` 的空间偏置与 Z 轴设计，不要拿“氛围感”糊过去。
6. 若视觉焦点或画外引导不清，先修 `shot_size_rhythm_preview`，再让 `镜头类型` 接着锁 POV 与 descriptor。
7. 若镜头类型开始承担器材、光影或运镜说明，先切回戏剧任务、角度家族、POV 与 descriptor 槽位，再继续汇流。
8. 若 review 难以判断 patch 是否成立，通常是 `thinking_process -> patch_payload` 的映射断了。

## Reusable Heuristics

- `分镜构图` 最稳的产物不是“漂亮描述”，而是“后三支拿到后不需要重判骨架”的 shot spine。
- 叶子顺序固定为 `构图形式 -> 景别景深 -> 镜头类型`；只要顺序倒过来，通常就会出现画面没站住却先谈距离或类型的漂移。
- `构图形式` 先回答“画面怎么站住”，`景别景深` 再回答“观众离多远”，`镜头类型` 最后回答“观众怎么被带着看”。
- `景别景深` 不只锁远近，还要锁“纳入多少环境、前中后景谁可读、观众被引向哪侧画外、第一眼看哪里”。
- `广 / 标 / 长` 在构图层只能被写成空间纳入或关系压缩的倾向，不能退化成器材词。
- 深景深适合同时读取前后景，浅景深适合锁单点，但失焦区绝不能误导叙事。
- 开放景框只有在明确画外方向与未知对象时才成立；否则只是信息缺失，不是悬念。
- 多人镜头先锁关系读法，再判断拉长 Z 轴还是压缩 Z 轴；别先写宽景再补理由。
- `构图形式` 不是 blocking 摘要；最稳的写法是“构图范式名 + 当前镜的几何任务”，例如“单点透视纵深构图，用消失点把压迫推向尽头”。
- `构图形式` 先过“留白/安全区”这一关，再谈风格名；连脸和关键动作都没摆稳时，讨论美学风格没有意义。
- 双人构图最常见的判断不是“有没有两个人”，而是“为什么这里该是 profile two-shot、front two-shot 还是 OTS”。
- `构图形式` 里写前景遮挡、框中框、负空间或荷兰角时，必须能回答它在做窥视、隔断、失衡还是悬置，而不是只提供表面样式。
- 构图层的“二维造三维”主要靠地平线、消失点、斜线、前中后景和遮挡关系，不靠镜头参数术语。
- `镜头类型` 是上位叶子名，不是单个 shot 字段；它内部应收束 `镜头类型 / 镜头框架 / 镜头视角` 等具体概念。
- 外部文章里的“镜头”往往是总类，不是当前叶子的专属名词；`close-up / wide / zoom out` 优先归 `景别景深`，`off-center` 优先归 `构图形式`，`镜头类型` 负责把这些上游决策继续协调到 POV、角度与 descriptor。
- `镜头角度` 可以直接视为“观众眼睛的站位”。只要角度说不清，角色状态、情绪落点和叙事立场就会一起发虚。
- `镜头类型` 最稳的写法不是“更有故事感”，而是“戏剧任务 + 角度家族 + POV 方向 + descriptor 槽位”。
- `过肩` 最适合先建立关系，`POV` 最适合把观众送进某一方眼睛；二者常常应该连续出现，而不是互相替代。
- `荷兰角` 应只在危险、异常、失衡、精神不稳或爆发前张力中局部使用；把它当默认“酷”角度会迅速削弱叙事可信度。
- `镜头类型` 不是器材，`景别景深` 不是光圈；一旦名词被误读，构图层就会越权到摄影层。
- `slot_id / 构图骨架 / 视线组织 / beat_refs` 是最小可交付集，缺任一项都说明构图骨架还没站住。
