## Repair Playbook

1. 先确认失败属于来源不足、画面基调缺失、九维缺项、地点清单污染、摄影越权、生成请求污染、因果链缺失、字数不合规还是报告证据缺失。
2. 若来源不足，回到 `N1-INTAKE`，不要用热门场景风格或随机环境词补空。
3. 若缺少画面基调，输出标记为 `tone_missing_candidate`；不得假装已经继承全局视觉协议。
4. 若参考图/视频带来具体地点或建筑污染，回到 `N3-TONE-REFERENCE-EVIDENCE`，只保留空间、层次、环境光、氛围介质和时代质地。
5. 若 prompt 出现场景清单、构图、摄影或生成请求，回到 `N6-PROMPT-DISTILL`，用场景层级抽象表达替换。
6. 若九维矩阵不完整，回到 `N4-SCENE-STYLE-DIMENSIONS`，逐项补齐并删除具体地点。
7. 若统一原则只是口号，回到 `N5-UNIFICATION-STRATEGY`，补至少 5 条可回指来源的映射。
8. 用户指出“脚本化/偷懒/未思考/未差异化”时，直接废弃候选协议回源层重做，不接受替换几个地点词或润色句子。
9. 修复后必须重新执行 `GATE-SS-01` 到 `GATE-SS-10`，并把失败码、返工点和复审结果写入 `Repair Log`。

## Reusable Heuristics

- 场景风格不是场景清单；它回答所有场景共享的空间秩序、环境密度和视觉统一性，而不是列出有哪些地点。
- 好的场景风格应能让后续场景设计师知道“空间怎么组织”，但仍保留具体地点、建筑、道具和布局的创作空间。
- `画面基调` 提供全局媒介和渲染纪律，`场景风格` 提供场景空间如何承接这种纪律；不要反向改写上游。
- 色彩系统可以在场景层写成“受控色彩纪律、明度层级、对比组织”，不要落成具体颜色清单。
- 光影可以写成结果和空间作用，例如体积雾、空气透视、漫反射扩散、层次衰减；不要写灯位、镜头或布光方案。
- 生态系统和人工/科技系统应写组织方式和密度，而不是写具体树种、设备、屏幕、车辆或道具。
- 氛围介质和粒子层最容易变成廉价氛围词；必须说明它们承担空间层次、时代包浆或环境压力，而不是单纯“更有氛围”。
- 约 100 字 prompt 的稳定结构：空间尺度/秩序 -> 空间光影 -> 色彩/明度纪律 -> 氛围介质/粒子层 -> 生态/人工系统组织 -> 环境层次/时代质地 -> 视觉统一性和负面边界。
- 如果一个词只能服务某个具体地点，它不属于场景风格；如果它能跨多个场景维持统一性，它才适合进入 `scene_style_prompt`。
- 单集场景风格可以承接本集空间压力、地点组合和情绪密度；不要覆盖整季场景风格基线。
