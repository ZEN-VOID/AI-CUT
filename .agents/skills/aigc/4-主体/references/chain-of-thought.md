# 4-主体思维链与字段主表

本文件承载 `4-主体` 父级路由层的字段主表、可见思维快照、工具后反思与返工入口真源。

## 模式与对象

| 项目 | 当前合同 |
| --- | --- |
| 默认模式 | `思维链优化设计`；若父级合同整体重建，可短暂借用 `原生思维链设计`，但最终仍回到本文件接口 |
| 目标对象 | `4-主体` 阶段级路由真源，服务 `projects/<项目名>/主体/subject-stage-index.md`、`validation-report.md` 与唯一下一入口 |
| 默认模型策略 | 若模型具备 reasoning / thinking 能力，优先高层路由判断 + 可见快照 + Gate；普通模型仅保留少量显式步骤支架 |
| 可见性边界 | 只保留可审计、可回写、可续跑的阶段裁决快照；不要求外显完整内部推理 |
| 工具 / 证据依赖 | 强依赖 `3-明细` 主文件、`2-组间` handoff、当前 `4-主体` 阶段状态，以及必要时各子路径的已有产物 |
| 受保护接口 | 不破坏 `FIELD-SUBJECT-01` 到 `FIELD-SUBJECT-04`；不把 `3-审计 / 4-面板` 误升格为默认主链，也不绕过 `1-清单 -> 2-设计` |

## Think-Think Design Snapshot

### 启发式工作链

- `不可删性`: 删掉这轮父级判断后，`4-主体` 会失去“先补主链缺口、再决定是否进入扩展链、最后给下游留唯一入口”的阶段协调能力。
- `最先推动`: 默认先推动 `FIELD-SUBJECT-01` 与 `FIELD-SUBJECT-02`；若主链优先级和当前命中子路径没锁住，禁止先谈落点美化或验收措辞。
- `先砍什么`: 先砍“审计/面板都是必经阶段”“哪个子路径最短就先走哪个”“目录已存在就视为阶段已完成”“主链没补完也能直接扩展”的伪判断。
- `真正比较尺`: 比较的不是哪条路最省事，而是哪种阶段路由最能补当前最大缺口、最少引入返工、最稳地保护主体真源主链。
- `立场纠偏`: 当前判断默认站在 `4-主体` 阶段运行时与下游 `5-画面 / 6-视频` 的消费立场，而不是站在单个子技能想扩权的立场。
- `工具后反思`: 读取 `3-明细 / 2-组间 / 当前阶段产物` 后，先重问主链是否完整、当前阻塞来自哪个子路径、是否要回上游补件，再继续路由。
- `落盘门`: 每条保留判断都必须落到 `FIELD-SUBJECT-*`、`Gate Summary`、`Fail Code` 或 `Rework Entry` 之一；落不到就删。

### 三轴三重裁决

| 裁决角色 | 在 `4-主体` 的具体问题 | 主要落点 |
| --- | --- | --- |
| `方向轴` | 当前阶段最先该补的是主链哪一站，还是已经满足进入 `3-审计 / 4-面板` 的条件 | `FIELD-SUBJECT-01`、`FIELD-SUBJECT-02` |
| `成立轴` | 这个路由判断是否被 `3-明细`、`2-组间`、当前主体产物状态与用户请求共同支撑 | `FIELD-SUBJECT-01`、`FIELD-SUBJECT-02` |
| `优选轴` | 在都成立的候选里，哪条路径最能减少返工、保持 canonical landing 稳定、并给下游留下唯一下一入口 | `FIELD-SUBJECT-03`、`FIELD-SUBJECT-04` |
| `硬门禁轴` | 是否出现绕过 `1-清单`、把扩展链误当主链、落点漂移、无唯一下一入口等强制否决条件 | `FAIL-SUBJECT-01` 到 `FAIL-SUBJECT-04` |

| 收敛层 | 本层先裁什么 | 服务 `field_id` | 本层收窄了什么 | 正确产物 |
| --- | --- | --- | --- | --- |
| `粗裁决` | `4-主体` 当前到底是在补主链缺口，还是满足条件后进入扩展链 | `FIELD-SUBJECT-01` | 收窄掉“所有子路径同权并列”“只按用户局部措辞跳转”的误路由 | 阶段前提与主链/扩展链语义 |
| `细裁决` | 当前命中的唯一路由是谁，四个子路径各自状态和职责是否清楚 | `FIELD-SUBJECT-02` | 收窄掉“目录存在即完成”“多个子路径同时主导”的模糊矩阵 | 子路径矩阵与当前命中结论 |
| `离散裁决` | 这轮判断最终落到哪里、如何结案、下一步由谁接手 | `FIELD-SUBJECT-03`、`FIELD-SUBJECT-04` | 收窄掉落点漂移、没有唯一下一入口、验收语焉不详的半闭环 | canonical landing + 阶段闭环 |

### 可见 / 隐藏分层

| 层级 | 保留内容 | 去向 |
| --- | --- | --- |
| `可见快照层` | 主链/扩展链语义、当前命中子路径、阶段落点、唯一下一入口与验收方式 | 进入字段表、阶段索引与验收文档 |
| `Gate Summary 层` | 当前阻塞、主导失败码、返工入口、confidence、unknowns、next_entry | 进入 `validation-report.md` 或阶段 sidecar |
| `隐藏推理层` | 被淘汰的子路径组合、未采用的捷径、过程中的局部比较废案 | 仅留在内部推理，不进可见合同 |

## 工具后反思与 Gate Summary

### 工具后反思

| 触发节点 | 必问反思 | 继续条件 | 不满足时回退 |
| --- | --- | --- | --- |
| 读取 `3-明细 / 2-组间 / 当前 4-主体 产物` 后 | 主体真源现在最缺的是清单、设计、审计还是面板；是否已有主链未补完却被局部诉求带偏 | 能明确回答“当前最高优先级未完成阶段是什么” | 回 `S1-S2` 重判 |
| 完成子路径初判后 | 当前路由是否唯一；是否只是因为用户提到了“审计/面板”就跳过主链 | 至少能说明为什么其他子路径不是当前主入口 | 回 `S1-S2` 收束 |
| 准备写 canonical landing 与阶段闭环前 | 当前落点是否与阶段真源一致；下一入口是否唯一且可执行 | 阶段根、子路径目录与下一入口全部可追踪 | 回 `S3-S4` 重写 |

### Gate Summary Contract

| 项目 | 必须回答 |
| --- | --- |
| `status` | `green / yellow / red` 三选一，不能省略 |
| `dominant_fail_code` | 当前最主要的失败码或 `none` |
| `rework_entry` | 若失败，必须指回 `S1` 到 `S4` |
| `confidence` | `0-1` 浮点值，说明当前阶段路由稳定度 |
| `unknowns` | 尚未被上游真源或子路径产物解决的缺口，最多 3 条 |
| `next_entry` | 当前建议进入的子路径或 `none` |

## Field Master

| field_id | 输出位置/字段 | 内容要求 | 默认责任 Step | 质量维度 | 失败码 |
| --- | --- | --- | --- | --- | --- |
| FIELD-SUBJECT-01 | 阶段前提 | 明确 `1-清单 -> 2-设计` 为默认主链，`3-审计 / 4-面板` 为按需扩展链，并说明当前为何命中该阶段语义 | S1 | 阶段语义清晰度 | FAIL-SUBJECT-01 |
| FIELD-SUBJECT-02 | 子路径矩阵 | 明确四个子路径的顺序、状态、职责、当前命中项与放弃其他路径的理由 | S2 | 路由完整性 | FAIL-SUBJECT-02 |
| FIELD-SUBJECT-03 | Canonical Landing | 固定 `projects/<项目名>/主体/` 及各子路径落点，不允许漂移到上游脚本层或下游资产层 | S3 | 落点稳定性 | FAIL-SUBJECT-03 |
| FIELD-SUBJECT-04 | 阶段闭环 | 能输出验收方式、当前阻塞与唯一下一入口；若暂不能继续，必须说明停在何处 | S4 | 可续跑性 | FAIL-SUBJECT-04 |

## Thought Pass Map

| step_id | 聚焦字段 | 核心问题 | 生成动作 | 未达标信号 |
| --- | --- | --- | --- | --- |
| S1 | FIELD-SUBJECT-01 | `4-主体` 当前到底解决什么，主链和扩展链谁优先 | 锁定阶段前提与当前主导语义 | 把审计/面板误写成主链，或忽略主链缺口 |
| S2 | FIELD-SUBJECT-02 | 当前请求应进哪个子路径，为什么不是另外几个 | 写子路径矩阵、当前命中项与放弃理由 | 有目录无路由，或多个子路径同时主导 |
| S3 | FIELD-SUBJECT-03 | 主体资产应落在哪里，哪些目录是 canonical | 固定阶段根与子路径落点 | 产物目录漂移，或把报告当主真源 |
| S4 | FIELD-SUBJECT-04 | 做完本轮后如何继续，谁来接手 | 写验收、阻塞与唯一下一入口 | 无法续跑，或只有“继续处理”空话 |

## Validation Matrix

| 检查项 | 通过标准 | 失败信号 |
| --- | --- | --- |
| 主链优先检查 | 主链未完成时，默认先命中 `1-清单` 或 `2-设计` | 主链缺口仍直接去 `3-审计 / 4-面板` |
| 路由唯一性检查 | 当前只有一个主命中子路径，且能解释放弃其他路径的原因 | 多个子路径并列成立，没有主入口 |
| canonical landing 检查 | 阶段根和子路径落点与 `projects/<项目名>/主体/` 保持一致 | 产物漂到 `3-明细`、`5-画面` 或临时报告目录 |
| 扩展链门槛检查 | 进入 `3-审计 / 4-面板` 前，已说明主链完成度与进入条件 | 扩展链被当成默认捷径 |
| 可见 / 隐藏分层检查 | 可见层只保留阶段裁决与闭环信息，不混入长篇过程废案 | 文档开始堆叠内部比较过程 |
| 工具后反思检查 | 关键读取节点后，发生过“是否继续 / 是否回退 / 是否改路由”的二次判断 | 上游结果一回来就机械继续 |
| 下一入口闭环检查 | `next_entry` 唯一，验收方式与阻塞状态可执行 | 只有“后续继续”之类空话 |
