# type-packs 扩维与调整指南

## Purpose

本文件是当前 `comic/type-packs/` 的维护 runbook。

当前真源已经改成：

- 根目录保留维护说明
- 题材知识真源集中在 `type-packs/漫画/`
- 当前运行时吸收逻辑以 `scripts/data_modules/comic_type_pack_resolver.py` 为准
- 根级动态配置真源为 `type-packs/runtime.yaml`
- 单题材动态配置真源为 `type-packs/漫画/<题材>/meta.yaml`
- 每个题材默认使用“单层题材目录 + 同名主文件 + 若干细化文件”结构

## Current Canonical Structure

### 1. 入口题材目录

形态：

```text
type-packs/漫画/少年战斗冒险/少年战斗冒险.md
type-packs/漫画/青春恋爱/青春恋爱.md
type-packs/漫画/情感关系剧/情感关系剧.md
type-packs/漫画/推理悬疑/推理悬疑.md
type-packs/漫画/黑暗奇幻/黑暗奇幻.md
```

规则：

- 目录名就是题材标识。
- 同名主文件是该题材入口知识文件。
- 后续如需补充细化，可继续在同目录增加更多 `.md`。
- 同目录 `meta.yaml` 负责该题材的动态匹配与阶段投影。

### 2. 题材内细化文件

形态：

```text
type-packs/漫画/少年战斗冒险/命名体系.md
type-packs/漫画/情感关系剧/关系原型.md
type-packs/漫画/推理悬疑/翻页悬念.md
```

规则：

- 这些文件不是新的题材节点，而是当前题材内部的结构化细化知识。
- 一个题材目录内应优先覆盖：
  - 核心冲突引擎
  - 角色/关系原型
  - 页面/分镜语法
  - 典型桥段库
  - 题材专属变体或钩子
- 不再单独拆出根层 “family craft”；跨题材技巧按各题材内部适配版本沉淀。

### 2.1 题材控制面字段

`meta.yaml` 现在不只承载 alias 和阶段投影，还必须承载一组更硬的经典漫画语法控制字段：

- `conflict_engine`
- `role_matrix`
- `page_turn_mechanism`
- `panel_grammar`
- `visual_carrier`
- `dialogue_register`
- `motif_system`
- `failure_modes`

维护原则：

- `stage_projection` 负责兼容现有下游链路。
- `control_surface` 负责更精确的题材语法控制。
- 若两者冲突，以 `control_surface` 作为题材判断的一手真源，再把需要下游消费的部分投影回 `stage_projection`。

### 3. 动态配置位

形态：

```text
type-packs/runtime.yaml
type-packs/漫画/<题材>/meta.yaml
```

规则：

- `runtime.yaml` 负责根级默认规则。
- `meta.yaml` 负责题材级别名、标签和阶段投影。
- resolver 每次任务执行都会重新扫描它们，因此升级或扩展题材包时会自动吸收最新版本。

## Reading Logic

针对 `type-packs/漫画/`，当前推荐读取顺序固定为：

1. 先读 `_shared/type-pack-loading-contract.md`
2. 再命中入口题材目录
3. 若目录下存在同名主文件，先读同名主文件
4. 再读同目录 `meta.yaml`
5. 再按目录内排序补读其他细化 `.md`
6. 最后吸收 `runtime.yaml`

默认题材命中规则：

- 若题材名与 `type-packs/漫画/<题材>/` 目录同名，除非特别说明，否则默认该目录就是入口。
- 若用户给的是旧 pack 名、旧市场标签或题材内技巧词，优先在对应题材 `meta.yaml` 的 alias 中做兼容回收。

## Fast Path Matrix

| 变更类型 | 先改哪里 | 条件性改动 | 一般不要先改 |
| --- | --- | --- | --- |
| 新增一个漫画题材 | `type-packs/漫画/<题材>/<题材>.md + meta.yaml` | 若要接入更多细化，再补同目录 `.md` | 根 skill 长文案 |
| 给某题材补细节 | 对应题材目录下新增 `.md` 或更新 `meta.yaml` | 若影响 resolver digest，再同步测试 | 直接改其他兄弟题材 |
| 调整跨阶段默认规则 | `type-packs/runtime.yaml` | 若 consumer 样例依赖旧值，同步改样例/self-test | 直接改 Python 硬编码 |
| 重命名题材目录 | 目录 + 全仓引用扫描同步 | resolver / 文档 / 示例一起改 | 只挪文件本体 |

## Standard Procedures

### 1. 新增入口题材

最短操作：

1. 新建目录：
   - `type-packs/漫画/<题材>/`
2. 新建同名主文件：
   - `type-packs/漫画/<题材>/<题材>.md`
3. 新建元数据文件：
   - `type-packs/漫画/<题材>/meta.yaml`
4. 至少写清：
   - `pack_id`
   - `aliases`
   - `semantic_tags`
   - `stage_projection`
   - `control_surface`
5. 再写清内容层：
   - 核心冲突引擎
   - 角色原型
   - 页面语法
   - 典型桥段
   - 禁写模式

### 2. 给题材补细化

最短操作：

1. 直接在题材目录中新增 `.md`
2. 文件名尽量直接表达用途，例如：
   - `关系原型.md`
   - `剧情模板.md`
   - `翻页悬念.md`
   - `标题钩子.md`
3. 题材内细化若承接旧“工艺词”，需要写成该题材专属版本，而不是回到根层再建一个并列 pack
4. 若新增的是经典漫画控制字段对应的细化材料，优先让文件名和字段一一对应，例如：
   - `翻页悬念.md` 对应 `page_turn_mechanism`
   - `镜头与表情.md` 对应 `visual_carrier`
   - `命名体系.md` 对应 `motif_system`

### 3. 调整动态运行时规则

最短操作：

1. 改根级默认规则时，优先更新 `runtime.yaml`
2. 改单题材投影时，优先更新对应目录 `meta.yaml`
3. consumer 已有测试或样例时，同步补回归

## Minimal Checklist

做完变更后至少自检：

1. 新题材是否已包在同名目录内
2. 是否存在同名主文件
3. `meta.yaml` 是否存在且字段齐全
4. 目录内是否已有足够支撑该题材的细化文件
5. 旧路径引用是否已全仓扫描
6. 根 `SKILL.md`、`_shared/type-pack-loading-contract.md`、本指南是否仍然说的是当前结构
7. resolver 和对应 tests 是否同步更新

## Bottom Line

当前 `comic/type-packs/` 的主口径不是多层 pack 分类，而是：

1. 先在 `漫画/` 下建单层题材目录
2. 放入同名主文件
3. 补 `meta.yaml`
4. 再用多个细化文件把题材做厚
5. resolver 在任务执行时动态扫描这些真源，生成最新 `type_pack_context`
