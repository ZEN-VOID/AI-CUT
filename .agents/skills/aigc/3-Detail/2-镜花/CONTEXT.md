## Repair Playbook

1. 先确认 `1-水月/第N集.md` 是不是唯一主输入，避免误用其他 grouped text。
2. 再核对 `Init / Global` 是否已被翻译成导演/摄影可消费的约束。
3. 再确认每组 `锚点` 是否明确写出了 `水月承接`，避免镜头语言脱离上游事实。
4. 若成稿没有镜头组织力，先重做 `分镜表现`，不要直接补摄影词。
5. 若有 shot spine 但画面发虚，优先补 `摄影美学`；若运动失衡，再补 `运镜手法`。
6. 若转场抢戏，先回退 `4-转场特效`，不要继续堆特效词。
7. 超字数时先删重复审美修辞，再删次级说明，最后只留主视觉收益。

## Reusable Heuristics

- `2-镜花` 的关键不是“写得更花”，而是“让 `1-水月` 具备导演和摄影的执行骨架”。
- `2-镜花` 的镜头语言组织必须回答一个前置问题：它究竟承接了 `1-水月` 的哪条动作、情绪、空间或关系信息。
- 编号模块最稳的用法是：顶层串行推进，分类内部并发思考，最终只汇成一段组级 `镜花` prose。
- 若 `切换 -> 插入 -> 构图` 没做稳，后面的摄影、运镜、转场都会变成漂浮注释。
- `切换` 最稳的顺序是 `候选节拍 -> 节奏档位 -> 镜数预算区间 -> 秒位窗口`；不要把句数、动作数或 `tail-hook` 直接折成镜数。
- `构图` 不要从“电影感”之类的美学词起手，而要先锁 `主陪背景 -> 重心/负空间 -> 构图方式 -> 焦点/观看路径 -> 描述子槽 -> 空间锚点`，这样后续摄影和运镜才接得住。
- `Init / Global` 最有效的进入方式不是标签，而是光色倾向、材料选择、空间压迫、镜头距离和节奏温度。
- `0-Init + 2-Global` 最适合作为“怎么拍”的约束，而不是“拍了什么”的事实来源。
- 摄影链最稳的顺序不是直接并发 `光影 / 色彩 / 质感`，而是先锁摄影底座、视觉控制线和策略优先级，再让叶子模块消费同一套中层约束。
- 运镜的价值不在“镜头动了”，而在“观众为什么此刻需要跟着它动”。
- 转场和特效默认应克制，只有当它们帮助情绪、时间或空间切换时才值得保留。
- `4-转场特效` 最稳的进入顺序是：先判问题类型，再比直切是否最强，只有直切不够时才继续写组内/组间/特效。
- 想压字数时，优先保留分镜标记后的核心动作、构图收益和一项最关键的摄影/运镜设计。
- `镜花` 正文必须像导演在和摄影、美术、后期说同一件事，而不是四类模块轮流发言。
- 节点包若只有 `module-spec.yaml`，久了容易把解释层塞回 schema；最稳的做法是用同目录 `module-guide.md` 承接 why、误用和审美尺度。
- `镜花` 想真正有导演感，关键不是每层都更满，而是先明确这组最该让观众感到什么，再让 route profile 决定谁主谁辅。
- 吸收 `3-Detail/references/运镜手法.md` 这类共享维度细则时，branch 节点应下沉“默认路线、汇流裁决、挑战边界”，leaf 节点应下沉各自的必答问题、误用和字段分权，避免共享细则只停在上层总文档里。
