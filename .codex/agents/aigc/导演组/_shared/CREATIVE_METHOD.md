# 导演组共享创作方法合同

## 0. 目的

本文件是 `.codex/agents/aigc/导演组/` 的共享方法真源。

它不负责重写父 skill 的 canonical writeback、`agents_plan + patch / note / report` handoff 或阶段边界，而是专门回答一件事：

- 导演组角色怎样从项目证据出发，做出更稳定、更具体、更可下游消费的创作判断。

## 1. 适用范围

适用于以下角色：

- `全局风格设计师`
- `类型化指导`
- `导演`

不适用于以下动作：

- 父 skill 的角色路由与最终写回
- 越权创建 `projects/<项目名>/3-Detail/第N集.json`
- 用单次即兴灵感替代项目级证据

## 2. 证据优先级

所有创作判断必须遵守以下证据梯度：

1. `硬证据`
   - `0-Init/north_star.yaml`
   - `0-Init/init_handoff.yaml`
   - `story-source-manifest.yaml`
   - `1-Planning/2-剧本/第N集.md`（若存在）
   - `1-Planning/3-分组/第N集.md`
   - `1-Planning/3-分组/执行报告.md`（若存在）
   - 现有 `2-Global/*.md`
2. `可推断约束`
   - 从题材、叙事目标、分组任务中可稳定推出的观看距离、节奏重心、禁区
3. `创作放大`
   - 在不违背前两层的前提下，对视觉重心、情绪组织、导演打法做具像化放大

硬规则：

- 不得把无证据支撑的风格修辞写成项目事实。
- 不得把 episode/group 局部感受升级为项目级总则。
- 若某条关键判断无法在 `硬证据` 或 `可推断约束` 中找到支撑，必须降低主张强度，必要时返回 `report`。

## 3. 共用决策顺序

三个角色都必须按以下顺序思考，而不是直接开始写槽位：

1. `锁定不变量`
   - 先找项目目标、题材边界、分组锁轴、显式禁令。
2. `抽取真正冲突`
   - 当前项目/本集/本组最值得导演系统解决的 tension 是什么。
3. `选择主导观看策略`
   - 决定“观众最该怎么被引导去看”，而不是先堆风格形容词。
4. `翻译成下游可执行指令`
   - 让 `3-Detail / 4-Design / 5-Image / 6-Video` 能消费，而不是只读到空泛态度。
5. `做反向自检`
   - 删掉无证据的大词、与上游冲突的推断、不能落地到 patch 的漂亮空话。

## 4. 质量门禁

### 4.1 全局风格门禁

合格的 `全局风格` 至少要回答：

- 为什么这是一套项目级稳定底座，而不是某一集的局部气氛。
- 观众应该以什么观看距离进入这个项目。
- 渲染、质感、光影、节奏各自稳定在哪里。
- 哪些表达属于禁区，为什么会伤害项目统一性。
- 哪些下游自由度允许变化，哪些不得漂移。

不合格信号：

- 只有“高级、电影感、唯美、史诗感”之类风格词。
- 只描述某一集当前情绪，没有项目级稳定项。
- 没写禁区或下游约束，导致后续任何方向都像“合理”。

### 4.2 类型指导门禁

合格的 `类型指导` 至少要回答：

- 主类型、副类型、混合公式分别是什么。
- 观众合同是什么，应该期待什么，不该期待什么。
- 节奏、揭示、情绪递送的导演打法是什么。
- 什么是错误类型信号，为什么不能滑过去。
- 证据不足时的保底表达是什么。

不合格信号：

- 只有题材标签，没有导演打法。
- 把多个类型并列罗列，却没有主副关系和权重。
- 没有负例或禁区，导致下游无法判断“不能怎么做”。

### 4.3 导演意图门禁

合格的 `导演意图` 每个命中组至少要回答：

- 本组真正的剧情任务是什么。
- 画面里最该看见什么，而不是“能看什么”。
- 角色状态怎样推进，情绪如何转弯。
- 空间与气氛压力如何服务组目标。
- 下游最值得放大的方向是什么。

不合格信号：

- 写成“增强张力、突出人物、营造氛围”这类空话。
- 只重复剧情梗概，没有导演筛选后的观看重点。
- 下游读完后仍不知道该放大哪一个镜头、表演或空间判断。

## 5. 候选比较法

当存在多种合理创作方向时，不要平均摊开，而要显式比较后只保留主路径：

- 候选 A 更适合什么
- 候选 B 会损失什么
- 为什么本轮选当前路径
- 未采纳方向是否需要在 `note` 中留痕

禁止把多个互相冲突的方向同时写进 canonical patch。

## 6. 低证据退化与阻塞规则

出现以下任一情况时，应降低主张强度或返回 `report`：

- 分组目标本身不稳定
- 上游风格要求互相冲突
- 类型证据不足以支撑明确裁决
- 组级任务不足以支撑具体画面重心

退化原则：

- 先保留稳定不变量，再减少高风险发挥。
- 可以给“保守版 patch + 阻塞说明”，不要用想象补齐事实空洞。
- `report` 必须指出缺什么证据、阻塞哪一步、补什么最有用。

## 7. Patch 写法要求

- patch 必须是“判断 + 约束 + 下游可消费信号”，不是散文段子。
- note 必须说明取舍与未采纳方向，不能只重复 patch。
- report 必须可执行，说明阻塞证据和返工入口。
- 若沿用旧文档，应优先增量更新稳定章节，而不是重抹一遍。

## 8. 共享反模式

以下模式默认视为失败：

- 用风格形容词堆出“看起来很懂”的文案
- 把项目级风格写成单集情绪板
- 把类型指导写成百科标签列表
- 把导演意图写成剧情摘要
- 在证据不足时继续强行给确定性判断
