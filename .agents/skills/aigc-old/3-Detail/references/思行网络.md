# 3-Detail Thinking-Action Network

## Node Spine

`3-Detail` 当前按知行合一口径运行，主干固定为：

1. `N0 锁定任务与唯一输出`
2. `N1 锁输入范围与知识域`
3. `N2 P1-分镜构图`
4. `N3 P2-角色表现`
5. `N4 P3-氛围表现`
6. `N5 P4-摄影表现`
7. `N6 P5/P6-运镜与转场`
8. `N7 P7-验收`
9. `N8 One-Shot Closure`

## Pass-to-Node Mapping

| 原 pass | 当前节点 | 真正职责 |
| --- | --- | --- |
| `输入锁定` | `N0 + N1` | 锁任务、锁 scope、锁 knowledge mode |
| `1-分镜构图` | `N2` | 先决定镜数、正文切分点与镜级骨架 |
| `2-角色表现` | `N3` | 把人物目的与表演压力写清楚 |
| `3-氛围表现` | `N4` | 把空间和空气压强写清楚 |
| `4-摄影表现` | `N5` | 把光色质和视觉重力写清楚 |
| `5-运镜手法` | `N6` | 让镜头运动服务骨架 |
| `6-转场特效` | `N6` | 只补最小必要衔接收益 |
| `7-验收` | `N7 + N8` | 跑校验、写报告、做唯一 closure |

## Why Composition First

`分镜构图` 在当前阶段先行，不是因为它最“电影化”，而是因为它先承担了三件结构职责：

1. 决定这一组到底切几镜。
2. 决定每镜承接哪一段组级正文。
3. 决定每镜的 `时间 / 剧本正文 / 主体锚定 / 分镜构图` 骨架。

因此在 `N2 / P1` 必须同时产出：

- `detail.分镜数`
- `分镜列表.<分镜ID>.时间`
- `分镜列表.<分镜ID>.剧本正文`
- `分镜列表.<分镜ID>.主体锚定`
- `分镜列表.<分镜ID>.分镜构图`

## Re-entry Rules

- 若发现 scope 或知识域判错：回到 `N1`
- 若发现镜数不成立：回到 `N2`
- 若发现镜级正文切错：回到 `N2`
- 若 `角色表现` 写成机位和镜头：回到 `N3`
- 若 `氛围表现` 只有形容词没有环境承载：回到 `N4`
- 若 `摄影表现 / 运镜手法` 反向改骨架：回到 `N2` 或当前节点
- 若 `转场特效` 喧宾夺主：回到 `N6`
- 若 `validation-report.md` 缺少 closure 四段：回到 `N7` 或 `N8`

## Minimal Evidence Chain

每个节点至少要能回答：

- 我现在判断的是哪一个问题？
- 我现在写的是哪一个字段对象或哪一个报告槽位？
- 它依赖前面哪些已锁字段？
- 它具体往哪一个 JSON / report 槽位写？
- 它写完之后，下一个节点为什么能够继续？
