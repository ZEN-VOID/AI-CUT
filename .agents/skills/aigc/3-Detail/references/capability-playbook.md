# 3-Detail Capability Playbook

本文件吸收原 `制作组` 的共享提示稳定性合同、共享创作质量手册与 16 个角色 agent 的跨角色共识。

它不是第二真源。`SKILL.md` 负责网络骨架、节点门禁与唯一写回，本文件只负责回答：

- 共享执行总则是什么
- 六个高复杂维度该读哪份细则
- 维度之间怎样交接、汇流、复核与返工

## 共享执行总则

1. 先服务镜头任务，再追求风格表达。
2. 先锁可见锚点，再写字段内容。
3. 只补当前能力链拥有的字段，不扩写成整镜总论。
4. 证据不足时保守退化，用 `note` 或 `report` 显式说明缺口。
5. 任何 finish 类能力链都不得反向定义 skeleton 主任务。
6. `structural_staging_engine` 内部默认按 `节拍/密度 -> 景别 -> 主陪背景关系 -> 构图布局 -> 构图方式 -> 镜头描述子槽 -> 焦点/观看路径 -> 空间/几何` 的顺序收束，不跳写中段判断。
7. `performance_engine` 内部默认按 `主轴 -> 个性/习惯/下意识 -> 可见信号 -> 叙事性行为 -> 眉眼/身体传神强化 -> 越权清理` 的顺序收束，不得直接从抽象情绪词跳到结果字段。
8. `camera_movement_engine` 若要探索“同目标更酷解法”，必须先固定表现目标不变量，再比较镜头变体，不得借机重写 core 维度事实。
9. `cinematography_engine` 必须先回看 `全局风格 / 类型指导 / 导演意图`，再写当前镜组的光位、光影推进与色彩心理；不能只从局部镜头词汇直接下结论。
10. 若命中 `knowledge-base/电影学院派/电影摄影`，必须把其转译为 `主光源 / 辅助光 / 逆光 / 照明类型 / 光影流动 / 色相 / 明度 / 饱和度 / 色温 / 色彩心理` 这类当前字段可消费的语言，不得直接抄术语。

## 参考锚点前置包

在进入 skeleton 与六个维度细则前，先生成一份 `reference_anchor_note`，至少回答：

1. 初始化预设里哪些元素必须被看见，而不是只停留在项目设定中。
2. 可以借鉴哪位大师的哪类表现手法。
3. 可以对照哪部作品的哪类桥段结构。
4. 这些参考如何被转译成可执行、可 merge 的具像化表达。

注意：

- 参考锚点只做方法借鉴，不做风格复刻。
- 任何“大师 / 作品 / 桥段”都必须落成 `景别 / 镜头属性 / 镜头框架 / 镜头类型 / 镜头视角 / 主陪背景关系 / 构图布局 / 构图方式 / 焦点 / 节拍 / 观看路径 / 运动 / 光色 / 衔接` 这类可执行语言。
- 若命中学院派知识库，只能把其作为证据层，不得覆盖当前项目真源。

## `shot_skeleton_engine`

- 负责字段：
  - `分镜ID`
  - `时间段`
  - shot coverage
- 着手方面：
  - 组级剧情任务
  - 信息优先级
  - 节奏断点
  - preset / lock-axis 保护
- 细步骤：
  1. 先按剧情任务切镜，不按“镜头好看”切镜。
  2. 再按信息优先级确定每镜主任务。
  3. 再分配时间段与组内覆盖关系。
  4. 最后检查是否越过 preset 锁轴或重排已锁镜序。
- 质量门禁：
  - 一镜只承载一个主任务。
  - 镜序不跳轴，coverage 可被后续链路直接消费。
  - `note` 需要说明保留与放弃的切法。
- 回退：
  - 上游证据不足时返回 `report`。
  - 多种等价拆法但证据不够时，选更稳定、更保守版本。

## 跨维度交接规则

1. `shot_skeleton_engine -> 分镜表现 / 角色表现 / 场景氛围`
   - 没有 skeleton，不得进入六个维度细则。
2. `分镜表现 -> 角色表现`
   - 角色关系、动作路径与视线变化只能建立在稳定主陪背景层级、构图关系与空间关系之上。
3. `分镜表现 / 角色表现 / 场景氛围 -> 运镜手法 / 摄影美学`
   - finish 维度不能反向重写 core 维度主任务。
   - `角色表现` 若生成了需要被特写、停顿或光影重点捕捉的眉眼/动作提示，应作为 side input 提供给 finish 维度，但不得借此把表演字段越权写成镜头设计。
4. `运镜手法 -> 摄影美学`
   - 摄影合成要吸收已确定的镜头运动，但不替代运镜决定。
   - 摄影链应在 `分镜表现` 的组级节奏与运镜路线基础上决定 `照明类型` 与 `光影流动`，而不是把每镜拆成互不关联的局部布光。
   - 运镜若存在“同目标更酷变体”，只能以比较 note 或 challenge note 形式上送，不能静默覆盖默认路线。
5. `转场特效`
   - 默认最后进入，只在跨镜衔接收益明确时补入。
6. `参考锚点前置包 -> 分镜表现`
   - 节拍、景别、镜头属性、镜头框架、镜头类型、镜头视角、主陪背景关系、构图布局、构图方式、焦点与节奏密度的设计应优先吸收已转译的参考锚点，而不是直接把作品名写进字段。

## 跨维度冲突裁决

1. `skeleton` 优先级最高。
2. `分镜表现` 负责镜头描述子槽、主陪背景层级、构图布局/方式、空间和观看路径，优先于 `角色表现` 对站位的微调建议。
3. `角色表现` 负责表演逻辑，优先于 `运镜手法` 的表现性冲动。
4. `场景氛围` 负责体感来源，不得覆盖 `摄影美学` 的具体视觉控制线、照明类型与色彩定夺。
5. `运镜手法` 与 `摄影美学` 冲突时，先保叙事清晰度，再保视觉统一。
6. `转场特效` 永远不能反向否决前述五个维度的主任务。

## `continuity_review_engine`

- 检查顺序：
  1. 字段齐全度
  2. 镜序可读性
  3. 空间连续性
  4. 动作因果
  5. 情绪推进
  6. 跨字段 patch 兼容性
- 门禁：
  - `PASS` 只在命中切片完整、阅读顺畅、没有明显断裂时给出。
  - `REWORK` 必须说明断裂位置与返工入口。

## `source_audit_engine`

- 审计顺序：
  1. single source
  2. schema
  3. lineage
  4. overreach
  5. writeback 边界
- 门禁：
  - `PASS` 仅当输出仍是唯一 canonical 根文件，且无第二真源和越权写回。
  - `REWORK` 必须指出违规路径、违规字段和应回到的规则源。
