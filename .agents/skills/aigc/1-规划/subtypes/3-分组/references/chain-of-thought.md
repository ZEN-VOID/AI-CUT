# 3-分组 · 思维链细则

## 模块定位

- 本文件是 `3-分组` 按最新 `think-think` 规范升级后的标准思维链模块。
- 主 `SKILL.md` 继续保留 `Field Master / Thought Pass Map / Pass Table` 作为执行主合同。
- 本文件负责把 `S1-S7` 重写为 `运行模式 + 启发式工作链 + 三轴三重 + 可见快照 + Gate Summary` 的稳定合同。
- 本模块默认只暴露可审计的决策快照，不要求外显完整内部推理全文。

## 设计立场与运行模式

### 设计立场

1. `隐藏推理层`：允许模型在内部比较候选边界、路由与回退，但不把完整内部推理当成交付物。
2. `可见快照层`：对外只保留能支撑审计、回写和返工的决定性快照。
3. `字段落盘层`：所有关键判断都要回到 `FIELD-GRP-*` 与对应产物槽位。
4. `Gate Summary`：最终必须能把 PASS/FAIL、失败码和返工入口压成可续跑结论。

### 运行模式判定

| 模式 | 触发条件 | 外显策略 | 禁止误用 |
| --- | --- | --- | --- |
| `推理优先模式` | 使用原生 reasoning / thinking 模型执行 `3-分组` | 先用启发式工作链收窄，再输出可见快照与字段落点 | 把 `S1-S7` 写成僵硬的逐字脚本 |
| `支架回退模式` | 普通模型、thinking 关闭或上下文很嘈杂 | 保留 `S1-S7` 明确支架，但每步只保留高价值判断 | 把流程描述写得比分组任务本身更长 |
| `工具反思增强模式` | 需要依赖量化投影、validator 或外部检索结果 | 每次关键结果返回后先做二次判断，再继续写回 | 工具返回后机械进入下一步，不复核路由/边界 |

硬规则：

1. 默认优先 `推理优先模式`，`S1-S7` 是可见支架，不是对内部推理的全文脚本化替代。
2. 若进入 `工具反思增强模式`，至少在 `S3` 和 `S7` 执行一次显式二次判断。
3. 任一模式都不得把完整 CoT 当成主产物。

## Think-Think Design Snapshot

### 启发式工作链

- `不可删性`：删掉这轮判断，下游会在哪一处重新猜 `主路由 / 组边界 / 依赖顺序 / 下一阶段入口`？
- `最先推动`：当前最该推进的是 `FIELD-GRP-ROUTE-02`、`FIELD-GRP-BOUNDARY-03` 还是 `FIELD-GRP-DEP-06`，而不是直接补表面文字？
- `先砍什么`：哪些“平均切块、按组数凑整、只看负载分、不看集边界”的候选必须先删掉？
- `真正比较尺`：这轮比较到底以 `预设服从`、`结构成立`、`量化门槛` 还是 `交接效率` 为最终比较尺？
- `立场纠偏`：当前判断是在替下游 handoff 降低返工，还是只是在替当前撰写者省事？
- `工具后反思`：若 `scene-duration-projection` 或 validator 返回了新证据，必须先重判哪一个字段，而不是直接补丁式前进？
- `落盘门`：这轮判断最终要落到哪个 `FIELD-GRP-*`；若落不到字段或模板槽位，就不应保留为显式合同。

### 三向三重裁决轴

| 裁决轴/层 | 本层先裁什么 | 绝不带进下一层什么 | 服务字段 | 正确产物形态 |
| --- | --- | --- | --- | --- |
| `粗裁决 / 方向轴` | 当前任务是否真的属于 `3-分组`，以及 `G1/G2/G3` 谁是唯一主路由 | 还没判阶段边界就直接切组；多路由并存不裁决 | `FIELD-GRP-INPUT-01`、`FIELD-GRP-ROUTE-02` | 输入范围、阶段归属、主路由决议 |
| `细裁决 / 成立轴` | 哪些候选边界既符合结构证据，又不违反量化真源与集边界合同 | 平均切块、只凭单一分数、跨集补洞 | `FIELD-GRP-BOUNDARY-03` | 候选边界、排除理由、量化复核 |
| `离散裁决 / 优选轴` | 在成立候选中，哪种切法最利于交接、并行和下游消费 | “都可以”式并列答案；看起来聪明但无法回写的说明 | `FIELD-GRP-PLAN-04`、`FIELD-GRP-FILES-05`、`FIELD-GRP-DEP-06` | 组表、组容器、依赖/并行性检查 |
| `全层 veto / 硬门禁轴` | 是否跨越 `1-分集` 集边界、突破 `hard_text_window`、长出 `第N组.md`、或缺失返工入口 | 带着已命中 veto 的方案继续润色 | `FIELD-GRP-BOUNDARY-03`、`FIELD-GRP-QA-07` | Fail code、返工入口、暂停/回退结论 |

### 层内自省

每次完成 `S2-S6` 后，至少要回答：

1. 为什么是这个结果，而不是相邻边界、另一条路由或另一种交接方案？
2. 如果不是这个结果，会不会得到更强的结构成立性或更低的下游返工成本？
3. 删掉当前保留项，下游会失去哪一个明确判断能力？
4. 如果把这一步改写成普通流程脚本，它为什么会比当前启发式合同更差？

若第 2 问答案为“会”，必须返回当前层重选，不得直接把旧结果带进下一层。

## 可见快照与隐藏推理分层

### 可见快照必须保留

| 可见块 | 最低内容 |
| --- | --- |
| `输入范围与阶段归属` | 输入文件、材料顺序、是否属于 `3-分组` |
| `主路由决议` | 唯一路由 + 放弃其他路由的原因 |
| `候选边界与排除理由` | 结构锚点、量化复核、排除说明 |
| `组表与组容器落盘` | 每组范围、锚点、依赖、并行性、下游入口 |
| `依赖与并行性检查` | `dependency_density`、串行/并行说明 |
| `Gate Summary` | PASS/FAIL、失败码、返工入口、unknowns |

### 不进入可见合同的内容

- 完整内部比较全文
- 所有被淘汰候选的逐条推理过程
- 与 `FIELD-GRP-*`、模板槽位或失败码无关的长篇感想

## 标准思维链主链

| step_id | 聚焦字段 | 核心问题 | 生成动作 | 必留可见快照 |
| --- | --- | --- | --- | --- |
| `S1` | `FIELD-GRP-INPUT-01` | 输入是否完整且顺序清晰 | 扫描 `north_star / init_handoff / 第N集.md / 结构材料`，形成输入清单并确认阶段归属 | `执行报告.md / 输入清单` |
| `S2` | `FIELD-GRP-ROUTE-02` | 当前应走哪条主路由 | 按 `G1 > G2 > G3` 做唯一裁决，并解释放弃其他路由的原因 | `执行报告.md / 路由决议` |
| `S3` | `FIELD-GRP-BOUNDARY-03` | 候选边界为什么成立 | 先列结构锚点，再做量化复核，保留候选与排除理由 | `执行报告.md / 候选边界` |
| `S4` | `FIELD-GRP-PLAN-04` | 组表是否可交接 | 将最终边界写成对应 `第N集.md` 的分组计划表 | `第N集.md / 分组计划表` |
| `S5` | `FIELD-GRP-FILES-05` | 组容器是否完整可续跑 | 为每个组补齐固定字段、组目标、结构锚点、量化指标、依赖与并行性说明 | `第N集.md / 组级容器` |
| `S6` | `FIELD-GRP-DEP-06` | 串并行关系是否清晰可执行 | 汇总 `dependency_density`、并行项、串行项与下游 handoff | `执行报告.md / 依赖与并行性检查` |
| `S7` | `FIELD-GRP-QA-07` | 当前结果能否结案 | 结合模板骨架与 validator 输出 PASS/FAIL、失败码和返工入口 | `执行报告.md / 验收结论与返工项` |

## 工具后反思与 Gate Summary

### 工具后反思

1. 若 `scene-duration-projection` 或其他量化结果显示候选组突破 `hard_text_window`，先回到 `S3` 重判边界，不得在 `S4-S5` 局部打补丁。
2. 若 validator 指出缺字段、缺锚点或长出 `第N组.md`，先回到对应 `S4-S6` 修合同落点，再重新验收。
3. 若新证据推翻当前主路由，应直接返回 `S2`，不得在旧路由内硬做补偿说明。

### Gate Summary 最低合同

| gate | PASS 信号 | 失败码 | 默认返工入口 |
| --- | --- | --- | --- |
| `输入门` | 输入清单完整、范围清晰、无跨集误入 | `FAIL-GRP-INPUT` | `S1` |
| `路由门` | `G1/G2/G3` 唯一且有证据 | `FAIL-GRP-ROUTE` | `S2` |
| `边界门` | 候选边界具备结构与量化成立性 | `FAIL-GRP-BOUNDARY` | `S3` |
| `组表门` | `第N集.md / 分组计划表` 可交接、无越权文件 | `FAIL-GRP-PLAN` | `S4` |
| `容器门` | 组级容器字段完整、可续跑 | `FAIL-GRP-FILES` | `S5` |
| `依赖门` | 串并行关系明确、`dependency_density` 可解释 | `FAIL-GRP-DEP` | `S6` |
| `闭环门` | PASS/FAIL、失败码、返工入口齐全 | `FAIL-GRP-QA` | `S7` |

## 字段落盘快照

| 思维链结果 | 默认落盘位置 | 最低证据 |
| --- | --- | --- |
| 输入范围与阶段归属 | `执行报告.md / 输入清单` | 输入文件或材料名、顺序、范围、阶段归属 |
| 主路由裁决 | `执行报告.md / 路由决议` | 主路由 + 放弃其他路由原因 |
| 候选边界与排除理由 | `执行报告.md / 候选边界` | 结构锚点、量化复核、排除说明 |
| 集内分组计划 | `第N集.md / 分组计划表` | 每组范围、量化字段、依赖、并行性、下游入口 |
| 组级容器 | `第N集.md / 各组章节` | 组目标、结构锚点、量化指标、依赖与并行性、下游建议 |
| Gate Summary | `执行报告.md / 验收结论与返工项` | PASS/FAIL、失败码、返工入口、unknowns |

## 使用边界

1. 若问题属于“应该怎么分”，先看本文件与 `references/type-strategies.md`。
2. 若问题属于“输出长什么样”，看 `references/output-template.md` 和 `templates/*.md`。
3. 若问题属于“执行顺序、回退或阶段协作”，看 `references/execution-flow.md`。
4. 若问题属于“量化为什么不成立”，回到 `references/scene-duration-projection.md`。
