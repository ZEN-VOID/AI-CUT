# 3-Detail Thinking-Action Network

## Fixed Order

`3-Detail` 的固定主链如下：

1. `输入锁定`
2. `1-分镜构图`
3. `2-角色表现`
4. `3-氛围表现`
5. `4-摄影表现`
6. `5-运镜手法`
7. `6-转场特效`
8. `7-验收`

## Why Composition First

`分镜构图` 在当前阶段先行，不是因为它最“电影化”，而是因为它先承担了三件结构职责：

1. 决定这一组到底切几镜。
2. 决定每镜承接哪一段组级正文。
3. 决定每镜的 `时间 / 剧本正文 / 主体锚定 / 分镜构图` 骨架。

因此在 `P1` 必须同时产出：

- `detail.分镜数`
- `分镜列表.<分镜ID>.时间`
- `分镜列表.<分镜ID>.剧本正文`
- `分镜列表.<分镜ID>.主体锚定`
- `分镜列表.<分镜ID>.分镜构图`

## Pass Re-entry Rules

- 若发现镜数不成立：回到 `P1`
- 若发现镜级正文切错：回到 `P1`
- 若 `角色表现` 写成机位和镜头：回到 `P2`
- 若 `氛围表现` 只有形容词没有环境承载：回到 `P3`
- 若 `摄影表现 / 运镜手法` 反向改骨架：回到 `P1` 或当前 pass
- 若 `转场特效` 喧宾夺主：回到 `P6`

## Minimal Evidence Chain

每个 pass 至少要能回答：

- 我现在写的是哪一个字段对象？
- 它依赖前面哪些已锁字段？
- 它具体往哪一个 JSON 槽位写？
- 它写完之后，下一个 pass 为什么能够继续？
