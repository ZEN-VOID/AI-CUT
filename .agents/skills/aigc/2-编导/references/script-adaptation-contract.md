# Script Adaptation Contract

## Canonical Purpose

`2-编导` 将 `projects/aigc/<项目名>/1-分集/第N集.md` 投影为 `projects/aigc/<项目名>/2-编导/第N集.md`。输出是上游正文的影视剧本化结构投影，不是改写版小说、概要、梗概或自由发挥剧本。

## Fidelity Rules

1. 不得压缩、摘要、删减剧情事实。
2. 不得自由改写剧情因果。
3. 不得重排上游事件顺序。
4. 除新增 frontmatter、`【剧本正文】`、场景标题与字段标签外，必须完整承接上游原文信息量。
5. 上游中适合作为角色主观经验、感知、判断或内在压力的描写，可在不删改原意的前提下投影为 `独白（角色）`、`内心独白（角色）`、`心理反应` 或 `表演提示`。
6. 画面字段允许剧本化改写，目标是让内容可拍、可分组、可服务表演与调度；改写不得删除事实、改变事件顺序或把动作改成解释性旁白。

## Screenplay Projection Definition

本技能中的“剧本化”不是把小说句子换成更像剧本的抽象描述，而是把上游信息投影为三类可执行材料：

- 可见：镜头能拍到的身体、表情、物件、文字、光线、空间、位置、运动和屏幕/黑板显影。
- 可听：现场或声音设计能听到的对白、旁白、机械提示、脚步、铃声、摩擦、尖叫、静默后的环境声。
- 可执行：演员、导演、美术、声音部门拿到后能直接做动作、站位、表演、道具或声音处理；尤其要把心理、潜台词和权力关系转成目标、阻碍、策略、停顿、视线、呼吸、手部动作、身体距离和道具关系。

因此，`画面`、`动作画面`、`对白画面`、`音效画面`、`旁白画面`、`系统画面`、`环境描写`、`道具特写`、`表演提示` 都必须优先具像化。它们不得停留在概念判断、主题解释、作者评论、比喻性结论或读者感受上。

## Concrete Visual Rule

画面字段必须回答“摄影机看见什么”，而不是“观众理解什么”。

禁止写入画面字段的内容：

- 抽象概念：`规则测试开始`、`压抑气氛继续`、`现实身份被覆盖`、`人数压力`、`底牌`、`幸运还是诅咒`。
- 解释性因果：`因为没有人敢缺席`、`第一次意识到人数本身就是规则`、`确认队友候选`。
- 作者判断或主题句：`这是一场小型审判`、`不做灾害奇观化`、`真正的规则`。
- 心理或感受替代：`不敢靠近`、`正面摊牌的爽快`、`像在等什么发生`。
- 纯章节/任务说明：`第二节课是课堂作业`、`规则测试开始`、`没有字数限制`。

修复方式：

- 抽象概念 -> 转成可见物：黑板文字、系统字、课桌、手指、目光、灯光、屏幕、道具状态。
- 心理判断 -> 转成身体执行：呼吸、眼神、停顿、手指、坐姿、退半步、避开视线。
- 比喻 -> 转成具体构图：人物与物件的位置关系、光线边界、前后景、遮挡、留白。
- 任务说明 -> 删除，或改为可见动作/道具，例如“白纸发到每张课桌，纸面中央印着题目”。

## Dramatic And Performance Projection

高质量剧本化必须把“发生了什么”继续投影成“这场戏如何被演员和导演执行”。该层只强化上游已有信息，不新增剧情事实。

必做转换：

- 场景功能 -> `entry_state / pressure_source / turning_point / exit_state`，再落入具体画面、声音、道具、表演或群像字段。
- 潜台词 -> 行为策略；例如试探、隐瞒、求证、施压、求同盟、拖延，必须通过停顿、避视、反问、道具动作、身体距离或声线变化呈现。
- 心理变化 -> 演员任务；至少给出角色目标、阻碍、策略或外显动作中的两项。
- 权力关系 -> 场面调度；通过站坐、高低、远近、门口/讲台/窗边占位、群体视线和道具归属表现。
- 沉默与反应 -> 可见/可听余波；不得用新增对白替代上游没有写出的潜台词。

`2-编导` 允许使用 `场面调度` 和 `表演提示` 强化导演可执行性，但不得写摄影越权信息，例如机位、景别、镜头运动、分镜编号或 `镜头语言预设` 字段。

## Dialogue Freeze

- 对白必须逐字保真。
- 不做润色、删改、同义替换、语序重排。
- 引号内不得混入动作描写。
- 动作、表情、停顿、空间反应全部下沉到对应 `*画面`、`角色动作`、`表情特写` 或 `表演提示`。
- 正文使用中文双引号。

## Scene Title Contract

场景标题统一使用阿拉伯数字编号 + 好莱坞标准剧本 slugline：

```md
### 场景1：内景 永夜私立中学二年级A班教室 - 日
### 场景2：外景 学校操场 - 夜
```

规则：

- 标题不得写成剧情摘要、动作 beat 或主题解释。
- slugline 由 `内景/外景 + 场所 + 日/夜` 组成。
- 同一集内完全相同的 slugline 必须沿用同一个场景编号。
- 不得因为规则宣告、角色入场、公告、能力觉醒等叙事 beat 变化另起场景号。
- 日/夜不同，或真实地点/空间范围变化，才可视为新场景。
- 连续承接同一 slugline 时，只允许首次出现标题；后续 beat 直接接正文。

## Minimum Scene Content

每个场景至少包含一条正式剧本画面字段，例如：

- `环境描写`
- `角色动作`
- `动作画面`
- `群像画面`
- `表情特写`
- `道具特写`
- `系统画面`
- `规则显影`
- `现实灾难画面`
- `对白画面`
- `独白画面`
- `内心独白画面`
- `旁白画面`
- `音效画面`

`动作画面` 只承载可拍摄的身体动作或空间运动，不得直接粘贴小说章节名、心理解释、背景说明、抽象判断或“没有人知道”一类叙述句。

## Frontmatter

逐集编导稿必须包含最小 frontmatter：

```yaml
---
项目名: <项目名>
集数: 第N集
stage: 2-编导
source_episode_path: projects/aigc/<项目名>/1-分集/第N集.md
output_path: projects/aigc/<项目名>/2-编导/第N集.md
adaptation_mode: faithful_screenplay_projection
dialogue_lock: true
audio_visual_pairing: required
slugline_policy: stable_by_location_time
peak_visual_policy: strengthen_existing_source_payoff
enrichment_mode: none
---
```

可按项目需要增加 `source_coverage`、`scene_count`、`review_status` 等字段，但不得把 frontmatter 变成剧情摘要替代正文。
