# Core Constraints

`story` 技能树跨阶段共享的章节创作硬约束真源。

禁止在兄弟 skill、局部 references 或 runbook 中平行复制一套“简化版 core constraints”；子技能只能做 projection，不得把本文件降格为可有可无的背景说明。

## Governance Hard Rules

- 先判 truth role，再判 stage。
- `planned`、`current`、`validated_actual` 必须显式区分，不得混写。
- 共享脚本与共享合同只保留一个 canonical source；兄弟阶段不得静默复制。
- `review` 不拥有 `validation_status` 判定权。
- `5-Loopback` 只在 `PASS + handoff granted` 时允许 actualization 写回。

## 三大定律

| 定律 | 规则 | 检查方式 |
| --- | --- | --- |
| `规划真源即法律` | 严格执行当前规划真源；优先 `Planning/全息地图.json` 的 chapter board / story spine / thread 债务，缺失时才回退兼容视图；不得靠“写起来更顺”擅自换戏。 | 起盘、节奏、张力、终验均需回对规划节点。 |
| `设定即物理` | 实力、招式、物品、身份、世界规则不得越过当前 canonical 设定。 | 写作前后检查 `Cards` / 设定索引 / 已验证正文。 |
| `发明需识别` | 新角色、地点、物品、规则若确实出现，必须能被后续 data/loopback 层识别，而不是模糊一笔带过。 | 正文中给出清晰名称、作用与辨识线索。 |

## 新实体处理流程

1. 写正文时直接写可读文本，不要求 XML 或占位标签。
2. 若出现新实体，必须让正文本身足够明确，便于后续抽取。
3. 若实体是否成立仍不确定，应在下游标记 `uncertain`，而不是在正文里写模糊影子设定。

## 章节约束分层

### Hard

- 本章必须可读：读者能回答“发生了什么 / 谁在做什么 / 为什么”。
- 本章必须存在清晰推进：问题、目标、代价、关系变化、信息变化至少一项可识别。
- 若上章有明确承诺、钩子或未闭合问题，本章必须回应；允许部分兑现，不允许完全失忆。
- 禁止占位正文：`[待补充]`、`[TODO]`、`...（省略）...` 等一律不允许。

### Soft

- 开头应尽早进入冲突、风险或强情绪，避免长距离空转铺垫。
- 未闭合问题或下一章期待锚点宜落在章末或后段，但不要求机械卡字数。
- 局面变化应有节奏感；可把 `800-1400` 字一个脉冲视为常见参考，而非硬编码。
- 微兑现频率按题材 profile 调整，不要求机械等间距。

### Style

- 对话尽量带意图：试探、回避、施压、诱导，而不是纯说明句。
- 避免连续大段纯解释；若必须解释，应拆成“信息 + 行动/反应”。
- 避免“回去休息了”式机械收尾；若使用平缓收尾，仍应保留未闭合期待。

## 爽点与节奏

- 爽点密度由题材与章型共同决定；过渡章允许低密度，但不允许“整章无收获”。
- 组合爽点与里程碑爽点用滚动窗口评估（如 `5` 章 / `10-15` 章），用于预警而不是逐章硬判。
- 连续同类型爽点达到 `3` 章时记为风险预警，优先通过类型变体或执行差异化修正。

## Strand 平衡警告

| 情节线 | 警告条件 |
| --- | --- |
| `Quest`（主线） | 连续 `5+` 章缺席 |
| `Fire`（感情线） | `>10` 章未出现 |
| `Constellation`（世界观） | `>15` 章未出现 |

## 对 Drafting 子技能的投影规则

- `1-单集叙事起盘`：必须把 `规划真源即法律` 作为 scene chain 的首门禁。
- `2-节奏优化`：必须把 Hard 约束当作不可破坏边界，把 Soft/Style 约束当作节奏调谐器，而不是只做字数压缩。
- `6-追读力强化`：不得靠发明新事故、新能力或新设定制造假牵引。
- `7-润色`：不得把 Hard/Soft 约束润掉，只能优化表达与自然度。

## 禁止事项

- `[待补充]`、`[TODO]`、`...（省略）...` 等占位正文。
- 主角、配角或世界规则突然获得正文未交代的新能力、新道具、新权限。
- 仅靠删减必要因果、跳过回应上章承诺、或用空泛热闹措辞伪造“节奏变快”。
