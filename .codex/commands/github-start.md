# /github-start

用于把当前工作区创建为新的 GitHub 仓库，并完成本地初始化、远程 `origin` 配置与首次推送。

## 何时使用

- 需要把当前工作区创建为新的 GitHub 仓库。
- 需要完成本地 Git 初始化、远程 `origin` 配置与首次推送。
- 需要把“仓库创建 -> 初始提交 -> 首次同步”作为可复用流程执行。

## 输入

- 仓库名称：默认可从当前目录名推导
- 可见性：`private` / `public`
- 是否允许创建或更新本地辅助配置文件

## Mandatory Gates

- 涉及外部可见状态变更前，必须先确认仓库名与可见性。
- 若检测到已存在远程 `origin`、已有提交历史、或目标仓库同名冲突，必须进入幂等分支处理。
- 未确认前，不执行不可逆或高影响操作。

## 标准流程

1. 检查 `gh` 是否安装、已认证，且当前用户具备建仓权限。
2. 检查当前目录是否已是 Git 仓库；若不是，则执行初始化。
3. 识别是否已存在 `origin`、默认分支、现有提交、已存在的 `LICENSE`。
4. 在确认仓库名与可见性后，执行 `gh repo create ... --source=. --remote=origin` 或等价幂等分支。
5. 若项目缺失 `LICENSE` 且流程需要标准许可协议，则创建或提示补齐。
6. 执行初始提交，推送到远程并建立本地分支跟踪关系。
7. 若仓库存在项目级链接配置约定，再更新对应配置文件。

## 验证

- `gh auth status` 正常
- `git status -sb` 显示工作区干净或仅剩预期改动
- `git remote -v` 中 `origin` 指向目标仓库
- `git branch -vv` 显示当前分支已跟踪远程分支
- 首次推送后，远程仓库可被 `gh repo view` 或网页访问确认

## 稳定经验

- “建仓”任务真正危险的不是命令本身，而是未经确认就改变了外部可见状态，所以确认门禁必须前置。
- 初始化流程应优先设计为可重入，而不是假设用户总是从全新目录开始。
- 首次 `git add` 之前，优先补齐 `LICENSE` 与最明显的本地噪音忽略项。
- `git init / commit / push` 这类会争用索引锁的关键步骤尽量不要并行编排；出现 `index.lock` 时先确认锁是否已释放。
