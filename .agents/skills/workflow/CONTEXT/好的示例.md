## Example: 批量成片路径清晰

```yaml
scenario: 批量生成 6 条 workflow 成片
good_path:
  - N1 记录 shared_asset_roots=["projects/素材/","projects/示例/"]，并声明只读消费。
  - N1 设置 batch_work_root="projects/output/<日期>/过程/<batch-id>/"。
  - N3 读取或创建 asset_usage_ledger.json，记录历史素材使用。
  - N5 为每条视频写 planned_usage 和 variation_axes。
  - N7/N8 完成 preview、visual contract、dialogue sync 和 render 验证。
  - N9 将最终 MP4 归集到 "projects/output/<日期>/成片/"，并把 final_path 回写 ledger/report。
why_good: 通用素材池、过程文件和最终成片三层分离，用户能直接查看成片，也能追溯过程证据。
```

## Example: 严格台词字幕

```yaml
scenario: 口播视频 final render
good_path:
  - N4 优先消费 SRT/VTT/ASR；没有转写时逐 cue 人工校时。
  - 每条 dialogue cue 包含 caption_type、start/end、spoken_text、script_anchor、sync_method、audio_anchor。
  - N7/N8 运行 scripts/validate_dialogue_sync.py --strict-final。
  - C3/C7 只有在 dialogue_sync_validation.json 为 pass 或明确 conditional 时才可继续。
why_good: 字幕验收依据来自音频主时钟，而不是段落比例或总时长估算。
```

## Example: 社媒广告画面不退化为 PPT

```yaml
scenario: 有工具录屏、素材视频和图片资产的社媒广告
good_path:
  - N3 建立 asset_evidence.json，区分 video_background、pip_asset、editorial_overlay。
  - N5 默认用视频/影像/工具实录承托背景层；图片背景只在单段有 fallback_reason 时使用。
  - 大字报写 6-14 字编辑提炼，不逐句复述主字幕。
  - PiP 按 cue 绑定图片或视频 segment，并记录 match_reason。
  - N7 运行 validate_visual_contract.py；失败时按 finding 回 N4/N5/N6。
why_good: 背景有动势，字幕有主时钟，PiP 和大字报有信息分工，机械门禁能拦住明显视觉错误。
```

## Example: 素材预处理目录正确使用

```yaml
scenario: projects/素材 已建立开头、收益、漫剧、大字报、工作流、引流、资产图、转场和核心关键词分支
good_path:
  - 空目录只作为素材补充计划，不写入 asset_evidence.json。
  - 只有真实文件、manifest 或本轮视觉复核进入候选素材证据。
  - 选材时优先按 cue 语义查找对应分支；需要旧素材时只从明确 archive 中读取，并记录来源。
  - 修改或搬迁通用素材池文件前先获得用户授权。
why_good: 预处理树提升检索效率，但不会让 workflow 虚构素材存在。
```

## Example: 文案音频同名配对

```yaml
scenario: projects/内容/文案 与 projects/内容/音频 已放入 50 条批量素材
good_path:
  - N1 扫描 `projects/内容/文案/*.txt` 和 `projects/内容/音频/*.mp3`，用文件 stem 建立 `script_audio_pair_map`。
  - `文案17.txt` 只和 `文案17.mp3` 组成一条 batch item；缺同名音频则记录缺口或请求 TTS/人工授权。
  - `BGM.*` 进入 BGM 候选池，不进入 `audio_clock`。
  - N4 在 `dialogue_alignment.json` 记录 source_script、source_audio 和 stem。
why_good: 批量成片的字幕、旁白、输出文件和 ledger 都能追踪到同一条内容真源。
```

## Example: 素材使用监控落盘

```yaml
scenario: 批量成片使用了 projects/素材/ 下的视频切片和资产图
good_path:
  - N3 先读取或初始化 projects/素材使用监控.csv，了解素材历史使用次数。
  - N5/N8 仍把每条成片的具体 source_file、segment_id、analysis_slice_id 和层级写入 asset_usage_ledger.json。
  - N9 在本地 final MP4 验证通过后运行 update_asset_usage_monitor.py，把实际使用汇总到 CSV。
  - CSV 只保留 素材名、文件路径、使用次数、使用程度 四列，使用程度为 全片 或 部分切片。
why_good: 用户能快速查看全局素材复用情况，同时详细片段证据仍保留在运行 ledger 中。
```

## Example: 有节奏的四层拼接

```yaml
scenario: 爆款口播广告成片
good_path:
  - N3 从开头素材、纯漫剧素材、工作流素材、收益素材、引流素材和资产图中建立真实候选证据。
  - N5 先写 hook_opening、content_body、private_traffic_cta 三段，再为每段声明 background_video、semantic_pip、dialogue_caption、editorial_overlay。
  - background_throughline 使用纯漫剧或等价影像证据拉通，mask=none。
  - content_body 覆盖或解释 comic_drama、tool_demo、revenue_proof 三个支点。
  - 大字报只写一个核心词或短句，不复述字幕。
  - N7 运行 validate_visual_contract.py --strict-social-ad 并保存 visual_contract_validation.json。
why_good: 先有节奏结构，再有素材层次，成片不会退化成逐句随机换素材。
```
