# Suspense Design Contract

本文件定义 `2-卷章` 三层规划共用的悬念设计机制。它不创建新的并列 skill，而是约束部级、卷级、章级在规划阶段控制信息差、隐藏真相、误导路径和揭秘节奏。

## Core Principle

悬念设计的核心不是“少写信息”，而是控制三类认知差：

- `truth_state`：故事世界里真实发生了什么。
- `reader_state`：读者在当前层级结束时被允许知道什么。
- `pov_state`：主角或当前 POV 角色在当前层级结束时知道什么、误判什么。

规划必须避免把 `truth_state` 直接完整交给读者。除非任务明确要求读者先知，否则不得使用上帝视角一次性解释核心真相。

## Three-Level Ownership

| level | required field | ownership |
| --- | --- | --- |
| `1-部级` | `整部悬念总设计` | 锁整书核心谜面、整书悬念池、读者认知曲线、主角认知曲线、卷级揭秘节奏、长线误导策略、多重悬念编排规则、禁止提前揭露与终局回收要求。 |
| `2-卷级` | `本卷悬念开关` | 承接整部悬念，决定本卷新增悬念、本卷悬念线程表、本卷隐藏项、允许露出的表层信息、本卷误导/疑阵、本卷揭秘项、延后压力、本卷悬念负载和章级约束。 |
| `3-章级` | `本章悬念开关` | 承接卷级悬念，决定本章读者可知、角色可知、本章悬念线程动作、隐藏项、误导/疑阵、揭秘项、只埋不揭项、章末悬念压力、本章悬念负载和正文禁区。 |

## Multi-Suspense Thread Model

多重悬念必须按线程治理，不能只写成一串自然语言谜团。每条悬念线程建议使用如下字段：

| field | meaning |
| --- | --- |
| `suspense_id` | ASCII 安全 ID，例如 `sus-main-identity`、`sus-vol2-rule`、`sus-ch15-danger`。 |
| `suspense_name` | 读者可理解的悬念名。 |
| `suspense_type` | `身份 / 动机 / 规则 / 关系 / 幕后事件 / 危机 / 世界观 / 资源 / 情感 / 选择 / 其他`。 |
| `priority` | `主悬念 / 次悬念 / 局部悬念 / 误导悬念`。 |
| `status` | `未开启 / 已开启 / 误导中 / 局部揭秘 / 已揭秘 / 延后 / 冻结`。 |
| `owner_level` | `book / volume / chapter`，用于判断谁有权开启、关闭或改写该线程。 |
| `reveal_window` | 预计在哪一卷、哪几章或哪一阶段揭示。 |
| `dependency` | 依赖的线索、伏笔、时间线事件、角色关系或任务节点。 |
| `reader_state` | 当前读者知道什么、误以为什么。 |
| `pov_state` | 当前 POV/主角知道什么、误判什么。 |
| `next_action` | 当前层级对该悬念执行 `开启 / 加压 / 误导 / 局部揭示 / 揭秘 / 延后 / 回收 / 冻结` 中哪一种动作。 |

## Multi-Suspense Orchestration Rules

1. 同一层级可同时存在多条悬念线程，但必须区分主悬念、次悬念、局部悬念和误导悬念。
2. 主悬念负责长期追读，次悬念负责中程牵引，局部悬念负责章/场内推进；不得让局部悬念抢走主悬念的终局解释权。
3. 每章建议最多重点操作 `1` 条主/次悬念和 `1-2` 条局部悬念；超过时必须说明悬念负载理由。
4. 多条悬念之间必须声明关系：`遮蔽 / 递进 / 互证 / 冲突 / 转移注意 / 红鲱鱼 / 并行无关`。
5. 误导悬念必须能在后续被解释，不能只靠作者藏信息制造假难题。
6. 已开启悬念必须有 `reveal_window` 或明确的延后理由；不得无限期悬空。
7. 任何 `已揭秘` 或 `局部揭秘` 都必须回指至少一个 `dependency`，避免事后补线索。
8. 若一章同时揭秘多条悬念，必须区分主揭秘和副揭秘，避免读者认知负载过高。
9. 若一章只加压不揭示，必须给读者一个微兑现或局部进展，避免悬念疲劳。
10. 悬念线程状态变更必须同步检查线索、伏笔、时间线、节奏和规避。

## Relationship To Existing Fields

| field | relationship |
| --- | --- |
| `故事编年史 / 本卷时间线 / 本章时间推进` | 锁世界内真实事件顺序。悬念字段决定这些事件何时对读者或角色开放。 |
| `hidden_events / parallel_hidden_events` | 是真实事件层的隐藏事件，不等于读者信息层。必须用悬念字段说明当前层级是否揭示、误导或继续隐藏。 |
| `整体节奏曲线 / 本卷节奏曲线 / 本章节奏曲线` | 控制阅读压力和释放强度。悬念字段控制信息释放的开关。 |
| `本章线索` | 写当前章可见信息推进，必须服从 `本章读者可知 / 本章揭秘的`。 |
| `本章伏笔` | 写未来回看成立的铺设与兑现，必须服从 `本章只埋不揭的 / 终局回收要求`。 |
| `规避` | 写禁飞区；若涉及提前剧透、上帝视角解释或假悬念，应回指悬念字段。 |

## Hard Rules

1. 三层规划都必须显式区分 `读者可知`、`角色可知` 与 `真实发生`，不得把真相说明误当作悬念设计。
2. `需要隐藏的` 必须具体到真相、身份、动机、规则、关系、幕后事件或因果链，不得写成“保持神秘感”。
3. `揭秘的` 必须说明本层揭示的范围，允许局部揭示，但不得无依据跳出上层认知曲线。
4. `整书悬念池 / 本卷悬念线程表 / 本章悬念线程动作` 必须能追踪 `suspense_id / priority / status / reveal_window / dependency / next_action`。
5. `误导/疑阵` 必须有可回溯依据；不得制造无法兑现、无法解释或纯靠作者隐瞒的假线索。
6. 章级正文约束必须明确禁止上帝视角解释被隐藏信息；drafting 只能写读者与 POV 当前可接触的信息。
7. 若采用读者先知、倒叙推理或多 POV 先揭模式，必须在对应悬念字段显式声明例外理由和张力来源。
8. 悬念开关不得替代线索/伏笔；它只决定信息的开放、隐藏、误导和揭秘边界。
9. 每次揭秘前，至少要能回指一个已铺设的线索、伏笔、行为异常、规则异常或时间线锚点。
10. 章末悬念压力必须来自未完成动作、认知缺口、关系风险、危险逼近或选择代价，不得只靠突兀断句。
11. 局部修订悬念字段时，必须回看同层线索、伏笔、时间线、节奏和规避是否同步。

## Review Gates

- 部级：能看出整书从谜面到终局回收的认知曲线，且 `整书悬念池` 中每条线程有 ID、类型、优先级、状态、揭示窗口和依赖。
- 卷级：能看出本卷哪些悬念线程开启、加压、误导、揭秘、延期，且章级能执行。
- 章级：能看出本章对每条命中悬念线程执行什么动作、正文允许写什么、不能说什么、尾钩压力来自哪里，且线索/伏笔不与悬念开关冲突。
