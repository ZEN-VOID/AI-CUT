# 2-Global 增量写回与兼容投影

本文件细化 `incremental-patch`、legacy projection 处理与 advisory 采纳策略。
若与主 `SKILL.md` 冲突，以主合同为准。

## Incremental Patch Policy

### patch scope

- `episode_scope`
  - 命中单集，只更新该集的 `第N集.json`
- `group_scope`
  - 命中单组或多组，只 patch 指定 `groups[].global`
- `field_scope`
  - 命中单字段，只 patch 对应字段，不重写同组其他已确认字段
- `advisory_scope`
  - 只因某条 advisory 采纳变化而改写相应字段与验收说明

### hard rules

1. `incremental-patch` 必须保留未命中 scope 的既有确认内容。
2. 若 patch 已触发项目级判断变化，必须重新跑 `N3-PROJECT-GLOBAL`，并同步说明其波及范围。
3. 若 patch 只影响组级判断，不得顺手重写项目级总则。
4. 任何 patch 都不得把 legacy Markdown 文件当输入真源。

## Legacy Projection Policy

新执行不得生成或更新 legacy Markdown。历史迁移任务若确需派生，必须满足以下条件：

1. `第N集.json` 已完成当前轮确认
2. 迁移任务明确需要 legacy 文件
3. legacy 内容可以逐项回链到 JSON 现有字段

legacy 典型文件：

- `全局风格.md`
- `全集类型元素.md`
- `导演意图.md`

legacy 硬规则：

1. legacy 文件只能复述与组织 JSON 已有事实，不新增业务判断。
2. legacy 文件不得早于 JSON 校验。
3. legacy 文件与 JSON 冲突时，一律回收为 JSON 为准。
4. 旧 `分组类型元素.md` 不再生成；组级类型信号由 `groups[].global.类型元素` 承载。

## Advisory Adoption Policy

对 `team.yaml` 命中的 advisory，必须给出以下裁决之一：

- `accepted`
- `partially_accepted`
- `rejected_with_reason`
- `not_applicable`

记录最少信息：

- `advisory_member_list`
- `advisory_synthesis`
- 采纳结果影响到哪些 `project_global.*` 或 `groups[].global.*`

## Common Failure Patterns

| symptom | direct_cause | immediate_fix |
| --- | --- | --- |
| 只改了 legacy Markdown，忘了回写 JSON | 把投影误当 canonical | 回到 `第N集.json` 先补真源 |
| 局部 patch 顺手覆盖全量 groups | 没有先锁 `group_scope / field_scope` | 回到 `N1-MODE-LOCK` 重新锁 scope |
| advisory 被写成 stage-end refine | 没锁消费阶段 | 把 advisory 回收为 `N2-N4` 前置输入 |
| patch 改了项目级总则，却没检查所有组的继承 | 没意识到 `project_global` 有扇出效应 | 重跑受影响组的继承校验与验收 |
